# 2026 03 30 REV4.1 Current Workflow

## Current phase
GitHub-transitioned workflow activation on top of the current fallback-first POC baseline. The technical milestone after repo alignment remains the first working slice of the long-lifecycle integrated testing and reporting system.

## Default work cadence
1. Choose the **pass type**.
2. Anchor the work to the best available **current truth**:
   - GitHub branch/commit + current-pointer docs when the intended baseline is pushed
   - newer local tree / uploaded candidate snapshot when the repo is behind
3. State what should **not** be overcommitted yet.
4. Execute the code/doc/spec pass.
5. Provide **exact validation commands** where validation is relevant.
6. Update current-truth docs, lessons, prompts, manifest/history, and handoff artifacts.
7. Commit and push the intended GitHub state.
8. Export sparse/full snapshot only when useful for archive, relay, rollback, or chat transfer.
9. Ask whether to use `project relay` or `chat retrospective` when a meaningful block has just completed.

## GitHub-first truth workflow
Use this precedence:
1. pushed local repo state on the intended branch
2. newer unpushed local state
3. uploaded sparse/full/candidate snapshot used for the current pass
4. older archived snapshots

A new or empty repo does not automatically outrank newer local truth. The workflow goal is to reduce the time spent with local truth ahead of GitHub.

## ZIP truth-establishment workflow
When a ZIP snapshot is uploaded:
1. use **Python first** to inspect the ZIP contents
2. read `OPEN_THIS_FIRST.md`
3. read `SNAPSHOT_INDEX.json`
4. read current-pointer docs
5. confirm manifest / modification history
6. then inspect the active code files

Do not rely on non-Python ZIP opening paths first when establishing project truth.

## Snapshot/relay workflow guidance
- Snapshots now serve **archive/relay/recovery** roles first.
- Full, sparse, and candidate snapshot scripts should generate or refresh:
  - `OPEN_THIS_FIRST.md`
  - `SNAPSHOT_INDEX.json`
  - `docs/CURRENT_ACTIVE_FILE_PATHS.md`
- Snapshot metadata should record repo identity and commit/branch when available.
- If a candidate snapshot is newer than the repo, treat it as immediate truth for the pass and then reconcile back into GitHub.

## Script validation guidance
Snapshot scripts should be validated on the target shell/runtime, not just assumed portable. Current lessons from this chat family:
- avoid `[System.IO.Path]::GetRelativePath` for Windows PowerShell 5.1
- avoid parameter-default logic that depends on `$PSScriptRoot`
- avoid `TrimStart` char-casting pitfalls for slash cleanup
