# 2026 03 30 REV3.7 Long-Lifecycle Testing and Reporting System

## Purpose
This document defines the long-lifecycle integrated testing and reporting system for pyAIGameEngine. The system is intended to support a long engineering life cycle with append-only archival truth, regenerable current reports, apples-to-apples comparisons, and provisional performance-versus-complexity forecasting.

## Design goals
- preserve fallback-first CPU-authoritative truth while broadening test discipline
- support both modular test execution and integrated test runs
- treat archival test data as durable source-of-truth artifacts
- regenerate current human-readable reports from archival test data
- support smoke, acceptance, regression, benchmark, simulation-credibility, and future AI/RL-facing tests
- minimize local terminal prompting while still allowing explicit parameter control

## Core architecture
The test system should be divided into six layers.

### 1. Test orchestration layer
Primary object: `IntegratedTestHarness`

Responsibilities:
- parse run configuration
- register and select test suites
- collect environment metadata
- run suites in defined order
- aggregate structured results
- write append-only archival records
- invoke report regeneration

### 2. Suite registry layer
Primary object: `TestSuiteRegistry`

Responsibilities:
- register suite implementations by stable IDs
- support grouped execution by category
- allow future independent or integrated suite execution

Planned initial categories:
- `smoke`
- `acceptance`
- `regression`
- `benchmark`
- `simulation_credibility`
- `adapter_rl`

### 3. Result capture layer
Primary objects:
- `TestRunRecord`
- `SuiteResultRecord`
- `MetricRecord`
- `ArtifactReference`

Responsibilities:
- normalize structured outputs from all suites
- capture pass/fail/skip/inconclusive states
- capture timing, parameters, and artifact references

### 4. Archival storage layer
Primary durable truth product:
- `test_run_archive`

Recommended storage model:
- append-only JSONL for immediate readability and robustness
- optional secondary binary/columnar mirrors later for scale or analytics

### 5. Comparison and analysis layer
Primary objects:
- `ComparisonEngine`
- `CompatibilityRuleSet`
- `ProjectionEngine`

Responsibilities:
- identify apples-to-apples comparison groups
- compare current run to prior compatible runs
- compute simple trend summaries
- compute provisional complexity-normalized projections

### 6. Reporting layer
Primary objects:
- `CurrentTestReportBuilder`
- `VisualizationBuilder`

Responsibilities:
- regenerate the latest human-readable modeler-facing report
- emit summary tables, plots, and snapshot references
- never become the durable truth source itself

## Execution model
A single integrated run should be able to:
1. load config
2. collect metadata
3. run one or more suites
4. append results to archival data
5. regenerate latest report products
6. emit a concise terminal summary with paths to artifacts

## Modularity rule
Each suite should be independently runnable, but all suite outputs must conform to the shared result schema so integrated reporting remains possible.

## Truth rule
- archival test data is append-only and durable
- current human-readable reports are replaceable/regenerable
- the latest report must always be reproducible from archival test data

## Initial implementation recommendation
Implement the system in three steps:
1. append-only archival schema + integrated harness skeleton
2. first benchmark/smoke/acceptance integration with current engine outputs
3. report regeneration and comparison/projection tooling
