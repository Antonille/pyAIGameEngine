# 2026 03 30 REV4.2 Model Interface Control

## Purpose
This document defines the current accepted contract between model families and the engine-facing AI boundary.

## Decided now
- Models should consume contract-facing observation blocks, not raw internal solver layout.
- Model scheduling remains family-based and should continue to use:
  - cadence
  - freshness
  - fallback
  - batchability
- The engine should remain agnostic to model architecture details.
- Model-family experimentation belongs in adapters/wrappers/control layers, not in the hot simulation core.
- The project should support future heterogeneous decision-makers around the same low-level contract:
  - scripted controllers
  - RL agents
  - pretrained non-learning models
  - variable-rate control families

## Temporary scaffolding
- The current baseline policy and minimal env path may use narrower assumptions than the final contract as long as they are documented and do not become the de facto public model API.
- POC-local helpers that simplify current rollout/testing are acceptable while they clearly terminate at the packet boundary.

## Deferred
- framework-specific model lifecycle APIs
- broad public model registration API
- distributed/many-world model-serving guarantees
- full GPU transfer contracts and staged inference infrastructure hardening

## Open questions
- Which model-family metadata should become mandatory in the first explicit contract draft?
- How much model capability description should live in docs versus discoverable runtime schemas?
- What minimum set of model-family control hooks is enough before richer inference backends are introduced?

## Required model-facing contract topics
### Decided now
Every model-facing contract should eventually specify:
- required feature blocks
- expected shape and dtype
- control cadence expectations
- freshness and fallback behavior
- packet output shape/semantics
- deadline expectations
- profiling/timing hooks surfaced to the adapter layer

### Temporary scaffolding
- Some of these fields may still be distributed across existing scheduler, feature-pack, and packet docs/code while the first explicit control-doc set is assembled.

### Deferred
- dynamic contract negotiation
- model-capability marketplace / plugin discovery
- generalized serving/batching beyond current family planning

### Open questions
- Which model-family contract should be formalized first as the exemplar for future families?
