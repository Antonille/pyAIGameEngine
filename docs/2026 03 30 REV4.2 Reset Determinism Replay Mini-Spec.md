# 2026 03 30 REV4.2 Reset Determinism Replay Mini-Spec

## Purpose
This document defines the near-term reset, determinism, and replay expectations needed to support trustworthy fallback-first AI/RL experimentation.

## Decided now
- Deterministic or tightly repeatable fallback-first CPU behavior is an explicit near-term target for:
  - fixed scenario
  - fixed seed
  - fixed action stream
- Reset behavior should be explicit and scenario-controlled.
- Seed handling must be first-class metadata in adapter-facing and benchmark-facing flows.
- Replay expectations should be documented before many-world/headless parallel rollout becomes an active design constraint.
- The test/reporting system should preserve enough metadata to support future replay validation work.

## Temporary scaffolding
- Perfect full-engine replay guarantees are not yet claimed.
- Some current reset/seed behavior may still be spread across POC-local helpers and adapters rather than consolidated into one replay-ready control module.
- The current adapter rollout path is acceptable as an early evidence source even if it is not yet a full replay harness.

## Deferred
- formal many-world replay contracts
- distributed or cross-host deterministic replay guarantees
- full snapshot restore guarantees for every future subsystem
- public replay API hardening

## Open questions
- What exact state bundle should be defined as the minimum replay-restore unit for the current phase?
- Which components are currently deterministic under fixed seed, and which still need explicit tightening?
- What tolerance threshold should distinguish “tightly repeatable” from “bitwise deterministic” for the fallback-first CPU path?

## Near-term target contract
### Decided now
For a fixed scenario + fixed seed + fixed action stream, the near-term goal is:
- same schedule counts
- same packet mode behavior
- same event counts
- same benchmark-family outcome class
- same or tightly bounded floating-point result behavior

### Temporary scaffolding
- Small floating-point differences may still occur; the goal is controlled repeatability before stronger claims.

### Deferred
- bitwise-identical guarantees across all future environments and hardware combinations

### Open questions
- Which existing validation suites should be extended first to assert replay-oriented behavior?
