# 2026 03 30 REV3.1 AI-RL Integration Strategy

## Purpose
This document defines how AI and reinforcement learning should be integrated into pyAIGameEngine without destabilizing the engine and simulation foundations.

## Core strategy
AI/RL remains **downstream of engine credibility**.
The engine should first provide:
- stable runtime schemas
- stable packet/action interfaces
- repeatable benchmark/verification behavior
- controlled scenario definitions

Only then should broader RL experimentation become a central development driver.

## Integration order

### Stage 1 — scripted / heuristic baselines
Start with:
- simple scripted policies
- deterministic or low-variance scenario checks
- packet/action behavior that is easy to inspect manually

### Stage 2 — AI family experimentation
Use the current family/cadence/freshness/batchability model to exercise:
- different feature packs
- different packet contracts
- stable train-time input/output assumptions

### Stage 3 — environment adapter design
Introduce a **Gymnasium-compatible adapter** once:
- current intended-use scope is documented
- core runtime behavior is verified for the experiment class
- scenario reset / step / observation / reward semantics can be expressed cleanly

### Stage 4 — RL experimentation
After adapter readiness:
- keep training and evaluation environments logically separated
- start with simple tasks and narrow reward structures
- progress from simple to complex environments deliberately
- preserve benchmarkable baseline policies for comparison

## Train / eval separation policy
When RL experiments begin in earnest:
- training runs should use versioned configs and seeds
- evaluation should use separate scenario/config definitions
- benchmark/performance runs should remain distinct from learning-progress evaluations
- environment/API changes should be versioned so model results remain interpretable

## Observation and action guidance
- Observations should come from **contract-facing engine representations**, not raw internal solver structures.
- Actions should enter through the packet/action boundary or a compatible wrapper, preserving the current engine-side interface discipline.
- Keep the environment layer thin; do not let RL convenience reshape core engine layout prematurely.

## Scenario progression guidance
Recommended order:
1. toy deterministic scenarios
2. repeatable benchmark-aligned scenarios
3. modest stochastic or adversarial scenarios
4. larger-scale or more complex training environments

## Current readiness assessment
Already in place:
- family-based AI scheduling direction
- feature-pack and packet/action boundaries
- repeatable validation commands
- benchmark-visible packet behavior

Not yet ready for broad RL work:
- environment adapter contract
- richer scenario catalog
- explicit train/eval artifact versioning in active use
- stronger simulation credibility documentation in day-to-day workflow

## Current recommendation
Do not widen RL implementation yet.
First finish the lifecycle, V&V, simulation-credibility, and artifact/versioning scaffolding now being added, then use those documents to guide the first environment-adapter planning pass.
