# 2026 03 30 REV4.1 Project Relay Procedure

## Purpose
`project relay` is the named procedure for handing off the project from one chat to another while minimizing loss of truth, status awareness, validated outputs, workflow discipline, and restartability.

## Required relay truth-establishment order
Use the best available truth source:
1. pushed GitHub repo state on the intended branch when current
2. newer uploaded candidate/full snapshot when the repo is behind
3. older archived sparse/full artifacts only for comparison or recovery

When a ZIP package is involved, use **Python first** to inspect it and then read:
1. `OPEN_THIS_FIRST.md`
2. `SNAPSHOT_INDEX.json`
3. current-pointer docs
4. manifest / modification history
5. active code files

## Typical outputs
A successful project relay should produce:
1. reconciliation summary
2. refreshed current-truth docs
3. refreshed current-pointer docs
4. refreshed relay/bootstrap prompts
5. refreshed `OPEN_THIS_FIRST.md` and `SNAPSHOT_INDEX.json` when packaging a snapshot
6. refreshed `docs/CURRENT_ACTIVE_FILE_PATHS.md`
7. updated manifest and modification history
8. clean relay-ready candidate snapshot ZIP when needed
9. next recommended prompt
10. explicit note about whether GitHub is current or the local/snapshot truth is still ahead

## Additional relay consistency checks
Before packaging the candidate snapshot or declaring repo truth current, confirm that:
- current-pointer docs reference the latest intended docs
- referenced docs exist
- repo identity and branch/commit state are recorded when relevant
- `OPEN_THIS_FIRST.md` and `SNAPSHOT_INDEX.json` exist at the ZIP root for packaged snapshots
- `docs/CURRENT_ACTIVE_FILE_PATHS.md` exists and points to real files
- docs do not claim code truth that is absent from the supplied candidate snapshot or intended repo baseline
