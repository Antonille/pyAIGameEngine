# 2026 03 30 REV3.8 AI-RL Integration Strategy

## Purpose
This document defines how AI and reinforcement learning should be integrated into pyAIGameEngine without destabilizing the engine and simulation foundations.

## Core strategy
AI/RL remains **downstream of engine credibility**.
The engine should first provide:
- stable runtime schemas
- stable packet/action interfaces
- repeatable benchmark/verification behavior
- controlled scenario definitions
- a thin adapter boundary for future RL-facing use
- long-lifecycle archival test data and regenerable reporting

## Integration order

### Stage 1 — scripted / heuristic baselines
Start with:
- simple scripted policies
- deterministic or low-variance scenario checks
- packet/action behavior that is easy to inspect manually
- repeatable archival recording of run metadata and outcomes

### Stage 2 — AI family experimentation
Use the current family/cadence/freshness/batchability model to exercise:
- different feature packs
- different packet contracts
- stable train-time input/output assumptions

### Stage 3 — environment adapter readiness
Introduce a **Gymnasium-compatible shape later**, but first build a project-local thin adapter that:
- wraps the current engine-side loop
- uses contract-facing observations only
- preserves the accepted action boundary
- supports explicit `train`, `eval`, and `benchmark` modes
- supports named baseline policies
- records seed and mode as first-class metadata

### Stage 4 — RL experimentation
After adapter readiness:
- keep training and evaluation environments logically separated
- start with simple tasks and narrow reward structures
- progress from simple to complex environments deliberately
- preserve benchmarkable baseline policies for comparison
- keep experiment/run artifacts reproducible and versioned
