# 2026 03 30 REV3.3 Project Relay Procedure

## Purpose
`project relay` is the named procedure for handing off the project from one chat to another while minimizing loss of:
- architecture knowledge
- status awareness
- validated outputs
- workflow discipline
- prompt artifacts
- lifecycle / V&V / simulation-credibility governance state
- restartability

## When to use it
Use `project relay` when:
- a meaningful architecture/doc/implementation block has been completed
- validated outputs have accumulated and need consolidation
- several sparse refreshes have occurred since the last full snapshot
- interface, governance, or control docs have changed substantially
- a chat is becoming long enough that continuity risk is rising
- a clean candidate snapshot is needed for the next chat

## Soft trigger policy
The assistant should suggest, but not force, a project relay after a meaningful block.
The user may choose to:
- relay immediately
- defer to the next milestone
- finish the current milestone before relaying

## Typical outputs
A successful project relay should produce:
1. reconciliation summary
2. refreshed current-truth docs
3. refreshed current-pointer docs
4. refreshed relay/bootstrap prompts
5. updated lessons learned and workflow docs as needed
6. refreshed manifest and modification history
7. clean relay-ready candidate snapshot ZIP
8. next recommended prompt

## Additional relay consistency check
Before packaging the candidate snapshot, confirm that:
- current-pointer docs reference the latest intended docs
- referenced docs exist with clean space-separated filenames in the package
- transient export artifacts such as percent-encoded duplicate filenames are not left behind in the candidate snapshot
