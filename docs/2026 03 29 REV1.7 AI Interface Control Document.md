# 2026 03 29 REV1.7 AI Interface Control Document

## Purpose
Define the boundary between the engine core and the **AI developer** role.

## Audience
AI developer:
- supplies model-family implementations
- consumes designated feature blocks / observation contracts
- returns actions, decisions, latent memory/state
- may define reward shaping logic
- should be able to experiment without requiring deep engine-solver knowledge

## Why this exists now
The current project already treats these as architectural truths:
- AI inference should not run for every agent every plank-tic
- repeated families should use fixed runtime schemas and feature-pack plans
- AI scheduling should operate on family, cadence, freshness, and batchability

That makes this the right time to document the boundary.

## Engine responsibilities toward this interface
The engine must expose:
1. **Model-family registration contract**
   - family id
   - cadence
   - freshness expectations
   - fallback policy
2. **Feature block contract**
   - block names
   - widths
   - semantics
   - freshness rules
3. **Action/output contract**
   - action block names
   - payload shape
   - actor indexing rules
4. **Objective / reward-signal contract**
   - canonical engine-side objective state
   - canonical event/reward signals
5. **Memory/state contract**
   - actor-local memory blobs or latent state slots if applicable

## Recommended reward-handling split
### Decision: hybrid reward architecture
Reward logic should be split as follows:

#### Engine owns:
- canonical objective state
- canonical event/reward signals
- standard measurable signals
- reference/default reward calculators for standard tasks (optional)

#### AI backend owns:
- reward shaping policy
- training-specific weighting
- experiment-specific reward composition
- memory-aware interpretation if needed

### Why hybrid
Fully engine-owned reward logic is too rigid for AI experimentation.
Fully AI-owned reward logic reduces reproducibility and portability across tasks.
Hybrid gives modularity without giving up standardized objective signals.

## Engine internals hidden from the AI developer
The AI interface should hide:
- collision solver internals
- field atlas implementation details
- narrow-phase/broad-phase mechanics
- renderer internals
- low-level backend scheduling machinery that does not affect the contract

## Primary contracts
### FeatureBlockContract
Defines one AI-consumable feature block:
- block name
- width
- dtype
- semantics
- freshness policy

### AIActionPacket
Defines one AI->engine action/update handoff:
- actor family
- actor indices
- action block name
- payload
- metadata

### RewardSignalPacket
Defines one engine->AI canonical signal:
- objective id
- event name
- scalar
- metadata

## Ownership boundary
The AI developer owns:
- inference model implementation
- memory/latent-state policy
- training loop behavior
- reward shaping logic

The engine owns:
- canonical simulation state
- canonical objective/event signals
- scheduling integration points
- feature extraction contracts
- action application semantics

## Near-term implementation guidance
Do now:
- define feature block contracts
- define action packet contracts
- define reward signal contracts
- keep scheduler + feature-pack planning moving forward

Defer:
- final GPU staging API
- final training harness split
- final independent AI package split

## Risks / cautions
- Do not expose raw solver internals as the AI contract.
- Do not let reward semantics live only in ad hoc notebooks/scripts.
- Keep actor-family indexing and feature-block semantics explicit and versionable.
