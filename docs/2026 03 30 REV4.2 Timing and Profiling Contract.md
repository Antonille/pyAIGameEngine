# 2026 03 30 REV4.2 Timing and Profiling Contract

## Purpose
This document defines the near-term timing/profiling contract that AI-facing adapters and future model families should align with.

## Decided now
- Timing data should be collected and surfaced without leaking model-specific logic into the hot simulation core.
- The contract should remain centered on stage timing and family-level scheduling behavior.
- Near-term timing topics to preserve:
  - observation build time
  - inference / AI scheduling time
  - packet/action apply time
  - physics time
  - event processing time
  - render-snapshot/report-facing overhead
  - reset time when reset plumbing is formalized
- Adapter-facing layers should surface timing hooks that map cleanly onto the accepted integrated benchmark/reporting system.

## Temporary scaffolding
- Current timing visibility is early and benchmark/report oriented rather than a final profiling API.
- Existing hooks are sufficient for accepted benchmark evidence but not yet a finalized public timing contract.

## Deferred
- full distributed profiling stack
- GPU transfer timing hardening
- many-world timing contracts
- framework-specific profiler integrations as part of the engine contract

## Open questions
- Which timing fields should become mandatory in the first adapter-facing profiling surface?
- Should reset timing be added first via the adapter or via the harness once reset behavior is more explicit?
- Which timing granularity is the minimum stable contract versus implementation detail?

## Current evidence shaping this contract
### Decided now
The accepted adapter-facing benchmark evidence shows AI/scheduling/packet-side work is already a meaningful integrated cost center. That makes timing visibility across the adapter boundary a real contract concern, not a future luxury.

### Temporary scaffolding
- Current stage names and exact placement may still evolve as the adapter refit proceeds.

### Deferred
- public profiler API hardening before the adapter boundary is tightened

### Open questions
- What is the cleanest place for the adapter to expose current stage timing without teaching the adapter too much about internal implementation detail?
