# 2026 03 30 REV3.4 Next Implementation Gate - Scheduler and Packet Acceptance

## Purpose
This document identifies the best next implementation gate to clear before reopening broader code work and defines the acceptance criteria, evidence, and artifacts required to clear it.

## Gate selection
**Selected next implementation gate:**
**Phase 3 Gate D — Scheduler and Packet Acceptance Gate**

## Why this gate comes next
This gate should come before competing next steps because it sits directly between the now-complete governance foundation and any broader widening of implementation scope.

It should come before:
- broader packet complexity work
- RL environment adapter work
- wider simulation validation classes
- larger benchmark-matrix expansion

### Rationale
1. **It stabilizes the contract-facing control boundary.**
   The project has already validated runtime-schema direction, multi-family scheduling, packet ownership visibility, and ownership-aware benchmark behavior. The next high-value step is to turn those visible behaviors into explicit acceptance checks rather than only benchmark-observed behaviors.
2. **It reduces rework risk for every later AI/RL step.**
   AI adapters, Gymnasium-style wrappers, and scripted baseline policies all depend on predictable cadence/freshness/fallback scheduling and predictable packet ownership/baseline semantics.
3. **It is the narrowest gate that meaningfully strengthens current claims.**
   Phase 2 governance work is in place. The next missing piece is not more governance documents; it is stronger verification of the current packet/scheduling contract.
4. **It respects the project’s current deferrals.**
   This gate does not require reopening PyBullet, changing authoritative CPU storage, or widening the project to full RL integration.

## Gate objective
Convert the current multi-family scheduling and packet boundary from “benchmark-visible and architecture-valid” into “explicitly accepted for the present prototype scope.”

## Acceptance criteria
The gate is considered cleared when all of the following are true.

### A. Scheduling acceptance criteria
- A documented acceptance checklist exists for:
  - cadence behavior
  - freshness behavior
  - fallback policy behavior
  - batchability behavior
- At least one targeted validation path confirms that these scheduling rules behave as intended for the current family set.
- The validation is not only inferred from integrated benchmark output; at least one contract-focused scheduling check exists.

### B. Packet ownership and baseline acceptance criteria
- Ownership-lane semantics are stated explicitly enough that a reviewer can answer:
  - who owns a packet lane
  - when a lane is superseded
  - what `baseline_generation_id`, `baseline_ownership_key`, `supersedes_generation_id`, and `supersedes_ownership_key` mean
  - when `replace`, `delta`, `no_change`, and `delete` are legal
- At least one targeted validation path covers the current packet modes.
- At least one negative or edge-case validation path is defined for later implementation, even if not yet fully automated.

### C. Packet-layer metrics acceptance criteria
- Current packet-layer metrics remain benchmark-visible and documented:
  - action packet count / bytes
  - cache hits / misses / invalidations
  - reuse ratio
  - no_change count
  - delete count
  - invalidate count
- The current ownership-aware benchmark subset remains recorded with workload context.
- Any change that alters packet churn or packet counts must be described against the current benchmark baseline.

### D. Artifact and evidence acceptance criteria
- Exact local validation commands are documented.
- The V&V Matrix is updated to reflect this gate as the next strengthening step.
- Modification history and project-state manifest are refreshed.
- A sparse snapshot is produced for the gate-clearing pass.

## Required validation evidence
To clear this gate, the next implementation block should provide:
- one or more contract-focused scheduling checks
- one or more contract-focused packet acceptance checks
- benchmark-visible packet metrics after the change
- an updated benchmark interpretation if behavior changes materially
- updated docs and a sparse snapshot
