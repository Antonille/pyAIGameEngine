# 2026 03 30 REV3.6 Phase 4 Gate A - Environment Adapter and Baseline Policy Readiness

## Purpose
This document defines the next implementation gate after Phase 3 Gate D. The goal is to prepare a thin, non-authoritative AI/RL-facing boundary without allowing adapter convenience concerns to reshape the core engine.

## Gate name
**Phase 4 Gate A — Environment Adapter and Baseline Policy Readiness**

## Why this gate comes next
Phase 3 Gate D already established that scheduler cadence, freshness, fallback policy, batchability, and packet-lane semantics are explicit and accepted for the current prototype scope. The next most valuable boundary is the first controlled AI/RL-facing wrapper around those accepted engine-side contracts, not deeper packet work.

This gate should come before:
- broader RL training work
- richer environment catalogs
- larger benchmark-matrix expansion for RL-facing behavior
- higher-complexity learned policy integration

because all of those depend on a stable, explicitly scoped environment boundary and on baseline policies that provide interpretable comparison targets.

## Adapter boundary shape
The first adapter should be **thin, contract-facing, and non-authoritative**.

### Core rule
The adapter must wrap the existing engine-side execution path. It must not become a second authoritative simulation pathway.

### Recommended first adapter object
`POC1EnvironmentAdapter`

### Recommended responsibilities
- translate scenario bundle + seed + mode into a controlled engine reset/configuration
- expose a Gymnasium-compatible shape later, but initially remain a project-local wrapper
- expose contract-facing observations only
- accept adapter-facing actions or a baseline-policy selection
- emit step results plus metadata/artifact fields
- keep train / eval / benchmark mode tagging explicit

### Recommended public surface
- `reset(*, scenario_bundle_id, mode, seed, baseline_policy_id=None, options=None)`
- `step(action=None)`
- `close()`
- `get_run_metadata()`
- `render_snapshot()` later when needed

### Observation rule
Observations must be derived from contract-facing engine representations. Do not expose raw solver arrays or internal SoA storage directly.

### Action rule
The adapter may accept:
- a compact adapter action structure, or
- a baseline policy selection that produces adapter actions internally

The adapter must not bypass the accepted engine-side action boundary.

## First baseline policy scope
Two deterministic, interpretable baselines should be in scope for the first adapter-facing experiments.

### 1. `baseline_hold_v1`
Purpose:
- steady / hold behavior
- deterministic reference point
- useful for smoke, reset, artifact, and observation stability checks

### 2. `baseline_reflex_v1`
Purpose:
- one simple reflexive policy using currently available contract-facing observations
- should remain intentionally simple and local, not learned and not globally strategic
- intended as the first comparison target for later learned policies

### Policy constraints
- baseline policies are adapter-facing, not authoritative simulation logic
- deterministic under fixed mode + seed + scenario bundle unless explicitly documented otherwise
- versioned and named artifacts

## Train / eval / benchmark separation requirements
The adapter path must distinguish execution intent explicitly.

### Train mode
- may permit stochasticity consistent with the scenario bundle
- should record seed and scenario bundle ID explicitly
- should not be treated as apples-to-apples benchmark evidence by default

### Eval mode
- should use fixed or explicitly listed scenario bundle + seed combinations
- should be intended for behavioral comparison of baseline policies and later learned policies
- should record stable outcome metrics for comparison

### Benchmark mode
- should remain tied to performance-oriented measurements
- should preserve existing benchmark discipline
- should record packet-layer and stage-level metrics when relevant
- should not be conflated with training quality or policy-learning progress

## Artifact fields required for Gate A
Each adapter-facing run must be able to report at minimum:
- `scenario_version`
- `adapter_version`
- `seed`
- `baseline_policy_id`
- `mode`

Recommended additional fields later:
- `scenario_bundle_id`
- `observation_contract_version`
- `action_contract_version`

## Acceptance criteria for Gate A
Gate A is cleared when all of the following are true:

1. A minimal, non-authoritative environment adapter prototype exists.
2. The adapter exposes explicit mode tagging for `train`, `eval`, and `benchmark`.
3. The adapter reports the required artifact/versioning fields.
4. The first two baseline policies exist:
   - hold / steady
   - one simple reflex baseline
5. The adapter uses contract-facing observations and the accepted action boundary.
6. The validation evidence demonstrates that the adapter path does not disturb current fallback-first benchmark and validation behavior.

## Validation evidence required
- explicit Gate A acceptance script or equivalent repeatable validation command
- at least one successful hold / steady rollout
- at least one successful simple reflex rollout
- visible mode tagging in outputs
- visible artifact fields in outputs
- re-run of current smoke benchmark and rigid-body/field validation showing no obvious regression in the existing validation path

## Required artifacts for gate closure
- adapter module
- baseline-policy module or registry
- Gate A acceptance script
- updated Lifecycle and Phase Gates
- updated Verification and Validation Matrix
- updated AI-RL Integration Strategy
- updated Artifact and Versioning Plan
- updated Current Workflow / manifest / modification history / next-step prompts

## Deferred items
Not part of Gate A:
- broad RL training loop implementation
- learned policy training infrastructure
- broad environment catalog
- multi-agent wrapper work
- PyBullet widening
- authoritative GPU-state changes

## Recommended next action
Reopen code only through an implementation pass that explicitly tries to clear Gate A with a thin adapter path and the first two baseline policies.
