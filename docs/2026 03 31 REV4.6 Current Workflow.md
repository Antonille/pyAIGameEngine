# 2026 03 31 REV4.6 Current Workflow

## Current phase
GitHub-first workflow active on top of the fallback-first POC baseline, with accepted archival testing truth, AI/RL boundary hardening underway, and a pruned active documentation surface now promoted into repo-truth.

## Default work cadence
1. Choose the **pass type**.
2. Anchor the work to the best available **current truth**.
3. State what should **not** be overcommitted yet.
4. Perform **pre-pass retrieval**:
   - read relevant recent lessons from `docs/CURRENT_LESSONS_LEARNED.md`
   - read open items from `docs/CURRENT_LESSON_ACTION_REGISTER.md`
   - state which truth tier is being used for the pass
5. Execute the code/doc/spec pass.
6. Provide exact validation commands where relevant.
7. Flash-capture lesson rows during the pass after failed validation, workflow breakdown, interface clarification, or unexpected benchmark/runtime results.
8. End with retrospective synthesis and concrete-destination assignment.
9. Commit/push intended GitHub state when local truth is ready.
10. Export sparse/full snapshots only when they materially help recovery, relay, or rollback.

## Active-surface discipline
- Keep repo root minimal.
- Keep docs root curated.
- Move superseded revision series into archive instead of leaving them mixed with current truth.
- Do not let temporary support files from sparse refreshes become durable repo clutter.

## Canonical local environment variables
Use only these names in docs, scripts, prompts, and validation commands:
- `path_pyAIGameEngine` = local repo root
- `path_pyEngine_scripts` = `scripts\windows` folder
- `path_revs_pyAIGameEngine` = revisions / snapshot ZIP folder

Do **not** use or mention historical aliases such as `path_zip_revs` or `path_rev_zip`.
If a required variable is missing, scripts should fail fast with a clear error instead of silently falling back to stale aliases.

## File-transfer hygiene rule
- For durable repo files, prefer zipped artifacts or downloadable files over chat copy/paste.
- If text must be pasted from chat, paste into an editor that shows line endings, verify EOL, then save before commit.
- Treat chat copy/paste as supporting transport, not as a byte-accurate source of truth.

## Line-ending control rule
- The repo root should carry a committed `.gitattributes` file.
- PowerShell and Windows shell scripts should be checked out as `CRLF`.
- Cross-platform source, docs, JSON, and markup should be normalized as `LF`.
- After adding or updating `.gitattributes`, run `git add --renormalize .` and review the resulting diff before commit.

## Required audits
### Current-pointer freshness audit
Before relay or authoritative full snapshot publication:
- verify touched `CURRENT_*` files point at the latest intended active docs
- verify the current next-step prompt matches the real highest-priority next pass
- verify stale entry points are updated or recorded as open risks

### Full+sparse reconciliation audit
Before declaring a new full snapshot authoritative:
- compare candidate full snapshot against repo state and later sparse deltas
- merge intentionally accepted sparse work into the full baseline
- record intentionally excluded items

### Line-ending / transfer audit
Before committing workflow docs, scripts, or pasted-text-derived files:
- verify `.gitattributes` exists and covers the touched file types
- verify the file was delivered by artifact or reviewed in an editor with visible EOL state
- verify the commit does not contain unexpected mixed-EOL churn
