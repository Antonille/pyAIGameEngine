## Research findings by topic
### 1. Runtime layout and simulation architecture
The research strongly supports continuing with **SoA-first hot-path design**. Data-oriented ECS systems emphasize contiguous per-field storage, batching, determinism/control, and easier vectorization for compilers.

Practical implication for this project:
- keep hot runtime fields in fixed contiguous blocks
- keep sparse or optional payload handles separate
- do not widen the authoritative runtime into dynamic per-entity field fetching

### 2. AoSoA question
AoSoA appears as a valid later optimization strategy, especially in HPC particle/simulation tooling, but it is not the right next move here.

Promote AoSoA from “interesting idea” to “candidate under evaluation” only if:
- hot-field count grows substantially
- cache behavior measurably degrades
- vectorization opportunities are clearly limited by the current layout
- benchmark matrix results justify the added complexity

### 3. AI scheduling and batching direction
The family/cadence/freshness/batchability direction remains well aligned with multi-stage inference research.

The most important follow-on insight is that batching mode itself should become a future experiment axis:
- continuous-like
- chunked-like
- mixed-like
- disaggregated-like

This does **not** mean implementing a large inference runtime now. It means the scheduler/boundary should be designed so that these modes can later be compared cleanly.

### 4. Action packets and delta semantics
The current action-packet bridge is directionally correct, but the next design gap is clearer after the sweep: delta-style packets need explicit baseline/generation semantics.

The next spec should define:
- reuse semantics
- supersede semantics
- invalidation semantics
- no-change semantics
- removal/delete semantics
- baseline or generation identifiers

### 5. Tool candidates
#### Recommended current default
- **NumPy + Numba** for the current CPU-authoritative fallback path

#### Plausible later
- **ONNX Runtime** once model-family serving contracts and cross-language deployment needs are more concrete
- **Pythran** or **Cython** for stable numeric kernels that stop benefiting from Numba
- **CuPy** after the CPU-authoritative runtime/schema/packet boundary is stable enough to define real GPU migration targets

#### Deferred
- immediate GPU-first ownership of the runtime
- premature conversion to a different runtime layout
- premature long-term lock-in to one AI serving backend
