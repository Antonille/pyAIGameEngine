# 2026 03 30 REV4.3 Model Interface Control

## Purpose
Define the current expectations for model-family interaction with the engine boundary.

## Decided now
- Model-family logic stays outside the hot simulation core.
- Model-facing staging buffers may normalize dtypes and pack feature blocks, but authoritative runtime storage remains SoA-oriented and field-appropriate.
- Model-family cadence, freshness, fallback, and batchability remain scheduler-controlled rather than implicit per-agent/per-step inference.
- External controllers and RL agents should be able to grow around the same low-level packet contract used by scripted or heuristic families.

## Temporary scaffolding
- Current mock inference payload generation inside the action bridge remains POC-local scaffolding for controlled testing.
- The minimal Gym env is not a model API; it is an adapter test path.

## Deferred
- Binding the project to one training/inference framework.
- Stable heterogeneous multi-model orchestration beyond the current family planning direction.
- GPU-first model-serving assumptions in the authoritative core.

## Open questions
- First stable named baseline-policy set for archival comparison.
- Which feature-block families should be marked required vs optional for the first model-interface milestone.
- Whether packet ownership metadata needs a shortened relay/debug presentation format in current reports.
