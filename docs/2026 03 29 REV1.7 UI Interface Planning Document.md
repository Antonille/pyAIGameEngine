# 2026 03 29 REV1.7 UI Interface Planning Document

## Purpose
Plan the future boundary between the engine core and the **UI developer** role.

## Audience
UI developer:
- handles render integration
- handles input/configuration/display
- may later handle audio and user-facing tooling
- should work against a stable view/control contract rather than solver internals

## Urgency
This interface should be planned now, but does not need full implementation yet.

## Planned engine responsibilities toward this interface
The engine should eventually expose:
1. **Render snapshot contract**
   - visible entity subset
   - transform snapshot buffer
   - material/appearance handles
   - view metadata
2. **Control/input contract**
   - user/player command packets
   - debug tool commands
   - camera/configuration requests
3. **Diagnostics/debug contract**
   - overlays
   - profiler summaries
   - selected state views
4. **Event/audio planning hooks**
   - not yet required
   - should be anticipated in event architecture

## Engine internals hidden from the UI developer
The UI interface should hide:
- solver stage internals
- AI scheduler internals
- collision/field implementation details
- training/reward internals

## Primary planning contracts
### RenderSnapshotView
Future engine->UI view contract:
- visible entity indices
- transform buffer reference
- metadata for presentation

### ControlIntentPacket
Future UI->engine control contract:
- source
- command type
- payload

## Recommended timing
Do now:
- document the intended boundary
- keep render snapshot semantics visible in architecture docs
- avoid leaking solver state directly into rendering code

Defer:
- independent UI module/package
- final render scene graph contract
- sound/audio subsystem
- user settings/config stack

## Risks / cautions
- Avoid tying UI directly to current POC renderer internals.
- Avoid exposing raw simulation storage to UI code.
- Keep render-facing subsets explicitly separate from full simulation state.
