## Workflow heuristics
- Use a clearly named **pass type** whenever practical: research, architecture/spec, implementation, documentation/interface, relay/handoff, or retrospective.
- Start from **current project truth** and explicitly preserve known-good constraints such as fallback-first execution and PyBullet deferral.
- Ask for or deliver **exact validation commands**, expected outcomes, and the next recommended prompt.
- Treat **integrated benchmark/runtime truth** as more authoritative than isolated microbenchmark wins.
- Prefer **interface/control docs and lightweight façades** before major package or boundary refactors.
- Keep **simulation-authoritative** structures separate from **interface-facing** representations.
- Capture both **recommended now** and **deferred / not yet** items so future passes do not overcommit.
- Once cadence-aware scheduling exists, benchmark summaries should include **cumulative run counts**, not only last-cycle observations.
- When a new family or bridge changes the integrated workload mix, re-check benchmark winner claims before carrying them forward.
