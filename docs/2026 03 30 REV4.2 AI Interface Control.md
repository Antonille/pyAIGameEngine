# 2026 03 30 REV4.2 AI Interface Control

## Purpose
This document defines the current accepted control boundary between the pyAIGameEngine hot simulation core, contract-facing AI staging, adapters, and higher-level experiment logic.

## Decided now
- The engine core stays low-level, fast, and authoritative.
- The authoritative observation path is **runtime-schema-driven feature blocks staged from SoA-authoritative state**.
- The authoritative action path is:
  1. action packets
  2. action-bridge application
  3. authoritative write into `state.action`
  4. downstream application/physics consumption
- Control cadence is a **derived layer** over the authoritative plank-tic loop.
- The engine should expose contract-facing observation/action interfaces without freezing itself around one external RL framework.
- Adapter, reward-shaping, curriculum, and task logic belong outside the hot simulation core.
- Contract docs must remain explicit about ownership, freshness, and versioning.

## Temporary scaffolding
- The current minimal Gym-compatible path is temporary scaffolding.
- Direct writes into `state.action` from the minimal env path are acceptable only as transitional behavior when clearly documented as bypassing the preferred packet boundary.
- The current adapter path is sufficient for early rollout/testing evidence but should not become the public AI API by inertia.

## Deferred
- Stable public RL API hardening
- many-world / headless parallel rollout as an active milestone constraint
- model-family-specific interfaces in the engine core
- framework-specific abstractions that would center the engine around Gymnasium or any one training stack

## Open questions
- Which exact packet/adapter surfaces should become the first code-facing stable contract modules?
- How much adapter-facing helper logic should be shared across scripted, RL, and pretrained-model controllers?
- When should a formal public API surface be declared, and what validation gate should precede it?

## Observation path control
### Decided now
- Observation staging should occur outside the hot simulation core, using runtime-schema-driven feature blocks extracted from authoritative SoA state.
- Observation contracts should specify:
  - block layout
  - dtype
  - semantics
  - ownership
  - freshness policy
  - copy/zero-copy expectations
  - versioning
- Feature-block extraction should support heterogeneous future decision-makers without engine-side model logic.

### Temporary scaffolding
- POC-specific observation builders are acceptable while they are aligned with the runtime-schema direction and do not become a permanent alternate truth path.

### Deferred
- hard public guarantees for all observation blocks
- GPU-oriented observation transfer contracts beyond what is needed for current adapter planning

### Open questions
- Which existing observation blocks should be promoted first into explicit control docs with stable names and semantics?
- What minimum set of observation blocks should every baseline controller be able to rely on?

## Action path control
### Decided now
- Action packets are the intended authoritative AI-to-engine contract surface.
- Action-bridge application is the correct place to translate packet semantics into authoritative state writes.
- Physics/application should consume authoritative state, not adapter-specific structures.

### Temporary scaffolding
- The current direct `state.action` path in the minimal env is temporary and should be documented as a shortcut, not a preferred contract.

### Deferred
- public action token/vector hybrid API hardening
- many-world batched action execution guarantees

### Open questions
- Which packet modes should be considered minimum supported contract modes for the first explicit stable adapter-facing boundary?
- What is the smallest stable action contract that serves scripted policies, RL agents, and pretrained non-learning models together?
