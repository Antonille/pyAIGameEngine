# 2026 03 30 REV4.2 GitHub and Snapshot Workflow

## Purpose
Keep the local repo, the GitHub repo, snapshot exports, and project-truth docs aligned.

## Canonical repo identity
- GitHub repo: `Antonille/pyAIGameEngine`
- Remote URL: `https://github.com/Antonille/pyAIGameEngine.git`
- Expected default branch: `main`
- Visibility: `private`

## Baseline-active rule
The repo baseline is now active and pushed.
Use this rule:
- if the intended baseline is committed and pushed, GitHub is the primary shared truth
- if the local tree is newer and unpushed, the local tree is immediate truth and must be pushed or explicitly snapshotted/reconciled
- if a chat uses an uploaded candidate snapshot that is newer than the repo, that snapshot becomes immediate truth for the pass until reconciled back into GitHub

## Standard stopping-point routine
1. run validation commands
2. update current docs and manifest/history
3. review `git status`
4. commit with a meaningful message
5. push the intended branch
6. optionally export sparse or full snapshot
7. if snapshots or uploaded run outputs were used in the pass, make sure the repo catches back up

## Branching and merge discipline
- Use `main` as the stable shared baseline.
- For risky or multi-step work, prefer short-lived feature branches over long-lived drift in the local working tree.
- Use pull requests or equivalent reviewable diffs when a change benefits from discussion or coordination.
- Prefer small, scoped commits with meaningful messages over large mixed-purpose commits.
- Merge only after validation commands for the affected scope have been run and recorded.

## Forking guidance
- Keep the authoritative project repo private and centrally coordinated until the intended milestone for wider collaboration.
- Use forks later for external experimentation or contributor isolation when the project intentionally opens to that model.
- Do not treat forks as a substitute for the current disciplined central-baseline workflow during this phase.

## Snapshot role from this point forward
Snapshots remain important, but their role is secondary:
- archive / rollback
- relay / chat transfer
- large sparse refresh handoff
- preservation of generated artifacts and result bundles
- recovery when local work is ahead of the repo

## Snapshot metadata fields recommended now
When practical, include:
- repo full name
- branch
- commit sha
- whether local work was ahead of the repo at snapshot time
