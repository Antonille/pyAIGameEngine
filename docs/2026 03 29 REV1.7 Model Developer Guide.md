# 2026 03 29 REV1.7 Model Developer Guide

## What this guide is for
This guide explains how to think about the engine from the **model developer** side.

You are responsible for defining:
- scenario content
- coordinate systems
- geometry
- actors/archetypes
- materials/properties
- objectives
- initial conditions

You are **not** expected to know:
- how the solver stores its hot-path arrays
- how AI is scheduled internally
- how collisions are solved
- how rendering is implemented internally

## Mental model
You provide the engine with a **scenario bundle**:
- what exists
- where it exists
- how it is configured initially
- what the objectives are

The engine evolves that scenario over time.

## Near-term best practices
1. Define content semantically, not in terms of SoA storage details.
2. Keep repeated object families explicit.
3. Keep coordinate-system intent explicit.
4. Use validation before handing content to the engine.
5. Treat payload-heavy/heterogeneous objects as special cases, not the default.

## Recommended conceptual parts of a scenario
- coordinate systems
- geometry definitions
- actor/archetype definitions
- property/material bundles
- field source definitions
- objective definitions
- initial conditions

## Coordinate-system guidance
Use coordinate systems to express model structure and frame relationships.
Do not assume model-side coordinate-system descriptions need direct access to solver internals.

## Properties/materials guidance
Think in layers:
- default material/property groups
- optional special property groups
- payload-heavy complex objects only where needed

## Geometry guidance
Prefer simple/analytic/compound shapes where possible.
Use more detailed voxel/scaled-voxel constructions only when the model truly needs them.

## Validation mindset
Ask:
- are coordinate references valid?
- are payload handles consistent with masks?
- are objectives well-formed?
- are initial conditions physically or logically sane?

## Near-term deliverable expectation
The near-term engine goal is a clear, validated scenario/model interface, not yet a full separate modeling application.
