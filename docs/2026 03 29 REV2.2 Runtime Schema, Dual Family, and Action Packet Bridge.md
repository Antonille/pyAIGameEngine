# 2026 03 29 REV2.2 Runtime Schema, Dual Family, and Action Packet Bridge

## Purpose
Document the second validation pass for the runtime-schema / feature-pack / transfer-planner boundary and the first lightweight action-output bridge.

## Scope of this pass
This pass intentionally stayed within the fallback-first CPU-authoritative architecture. It did not introduce real model execution, GPU transfer code, or PyBullet-dependent logic.

## Validated additions
### 1. Second scheduled family
Added `agent_contextual_v1` to exercise the same `agent_family` runtime schema with:
- slower cadence
- looser freshness
- richer feature width
- different preferred batch size
- different output contract

### 2. Action-block schemas
Added a lightweight action bridge with two output contracts:
- `action4_v1` = primary direct action packet (`replace` mode)
- `action4_delta_v1` = contextual overlay packet (`delta` mode)

### 3. Versioned action packets
Transfer-planned batches now produce versioned action packets with:
- protocol version
- source model family
- actor/runtime family
- action block name
- actor indices
- payload rows
- stage deadline
- metadata describing batch/application context

### 4. Packet application boundary
The authoritative solver state still owns the `state.action` buffer, but packet application now happens through a contract-facing bridge layer. This keeps the packet boundary separate from raw solver internals.

## Current interpretation
This pass successfully validated:
- more than one scheduled family on the same runtime family
- more than one feature-pack width/profile
- more than one action output mode
- packet emission and packet application summaries in benchmark output

## What was intentionally deferred
- real model inference backends
- packet serialization/IPC
- explicit packet cache / reuse policy objects
- packet invalidation and deadline expiration logic
- GPU staging or DMA-style transfer planning
- generalized multi-runtime-family registration beyond the current `agent_family`

## Recommended next implementation direction
Make fallback policy and output reuse visible at the packet layer, then validate a second action-block contract or action result schema so packet application rules are not limited to the current 4-float action shape.
