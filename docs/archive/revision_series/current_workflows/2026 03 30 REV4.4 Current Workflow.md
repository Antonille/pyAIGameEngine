# 2026 03 30 REV4.4 Current Workflow

## Current phase
GitHub-first workflow active on top of the fallback-first POC baseline, with accepted archival testing truth in place, AI/RL boundary hardening underway, and a new lightweight lesson-action enforcement layer added.

## Default work cadence
1. Choose the **pass type**.
2. Anchor the work to the best available **current truth**.
3. State what should **not** be overcommitted yet.
4. Perform **pre-pass retrieval**:
   - read the most relevant recent lessons from `docs/CURRENT_LESSONS_LEARNED.md`
   - read open items from `docs/CURRENT_LESSON_ACTION_REGISTER.md`
   - explicitly note whether repo-truth, snapshot-truth, or only supporting local evidence is being used
5. Execute the code/doc/spec pass.
6. Provide exact validation commands where relevant.
7. **Flash-capture** lesson rows during the pass after:
   - failed validation
   - tooling/workflow breakdown
   - interface-boundary clarification
   - unexpected benchmark/runtime result
8. End the pass with **retrospective synthesis**:
   - update current docs, lessons, prompts, and manifests as needed
   - move accepted items into one concrete destination
   - keep open risks visible if they are not yet resolved
9. Commit and push the intended GitHub state when local truth is ready.
10. Export sparse/full snapshots only when useful for archive, relay, rollback, or recovery.

## Required workflow audits
### Current-pointer freshness audit
Before `project relay` or before declaring a new full snapshot authoritative:
- verify that touched `CURRENT_*` files point at the latest intended revisioned docs
- verify that the current next-step prompt matches the real next highest-priority pass
- verify that stale entry points are either updated or explicitly left as open risks

### Full+sparse reconciliation audit
Before any new full snapshot is declared authoritative:
- compare the candidate full snapshot against the repo state and newer sparse deltas
- confirm that important sparse-pass changes are merged into the new full baseline
- record any intentionally excluded sparse artifacts
- do not declare the new full snapshot authoritative until this check is complete

## Lesson-action usage rule
Use `docs/CURRENT_LESSON_ACTION_REGISTER.md` as the thin implementation layer on top of narrative lessons.
- `lesson_learned` = raw learning from a specific event
- `best_practice` = reusable rule validated by successful application
- `defer_rule` = explicit not-yet guidance
- `open_risk` = unresolved issue to revisit

Do not promote a lesson into `best_practice` until there is validation evidence.

## Concrete-destination rule
Every accepted lesson should land in one concrete destination:
- script/tool fix
- workflow checklist change
- current-pointer doc update
- prompt-template update
- validation/test addition
- deferred/not-yet list
- backlog item for later architecture work

## Relay trigger guidance
Ask about `project relay` after a meaningful architecture/doc/implementation block, especially after:
- validated outputs and doc refreshes
- several sparse refreshes
- interface/control document changes
- milestone completion or near-completion

This remains a soft question, not a forced stop.
