# 2026 03 30 REV4.0 Project Relay Procedure

## Purpose
`project relay` is the named procedure for handing off the project from one chat to another while minimizing loss of truth, status awareness, validated outputs, workflow discipline, and restartability.

## Required relay truth-establishment order
When a candidate/full snapshot is available, establish truth in this order:
1. `OPEN_THIS_FIRST.md`
2. `SNAPSHOT_INDEX.json`
3. current-pointer docs
4. manifest / modification history
5. active code files

Use **Python first** to inspect the ZIP package.

## Typical outputs
A successful project relay should produce:
1. reconciliation summary
2. refreshed current-truth docs
3. refreshed current-pointer docs
4. refreshed relay/bootstrap prompts
5. refreshed `OPEN_THIS_FIRST.md`
6. refreshed `SNAPSHOT_INDEX.json`
7. refreshed `docs/CURRENT_ACTIVE_FILE_PATHS.md`
8. updated manifest and modification history
9. clean relay-ready candidate snapshot ZIP
10. next recommended prompt

## Additional relay consistency checks
Before packaging the candidate snapshot, confirm that:
- current-pointer docs reference the latest intended docs
- referenced docs exist with clean filenames in the package
- `OPEN_THIS_FIRST.md` and `SNAPSHOT_INDEX.json` exist at the ZIP root
- `docs/CURRENT_ACTIVE_FILE_PATHS.md` exists and points to real files
- docs do not claim code truth that is absent from the candidate snapshot
