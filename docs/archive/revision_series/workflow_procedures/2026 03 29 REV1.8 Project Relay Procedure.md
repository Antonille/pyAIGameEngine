# 2026 03 29 REV1.8 Project Relay Procedure

## Purpose
`project relay` is the named procedure for handing off the project from one chat to another while minimizing loss of:
- architecture knowledge
- status awareness
- validated outputs
- workflow discipline
- prompt artifacts
- restartability

## When to use it
Use `project relay` when:
- a meaningful architecture/doc/implementation block has been completed
- validated outputs have accumulated and need consolidation
- several sparse refreshes have occurred since the last full snapshot
- interface or control docs have changed substantially
- a chat is becoming long enough that continuity risk is rising

## Soft trigger policy
The assistant should suggest, but not force, a project relay after a meaningful block.
The user may choose to:
- relay immediately
- defer to the next milestone
- finish the current milestone before relaying

## Inputs
Helpful inputs include:
- latest local or uploaded snapshot ZIP
- latest sparse snapshot ZIPs not yet merged locally
- latest validation output
- any manual local edits not yet documented

## Required outputs
A successful project relay should produce:
1. reconciliation summary
2. refreshed current-truth docs
3. refreshed current-pointer docs
4. updated prompts/templates
5. updated lessons learned and workflow docs
6. clean full snapshot ZIP
7. next-chat bootstrap prompt
8. next recommended prompt

## Key constraints
- preserve fallback-first architecture unless validated work changes it
- preserve latest validated benchmark/runtime truth
- prefer restartability and clarity over brevity
- exclude transient artifacts from full snapshot packaging
