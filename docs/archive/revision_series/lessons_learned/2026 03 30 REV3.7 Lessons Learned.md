# 2026 03 30 REV3.7 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, prompting, and now long-lifecycle test/reporting lessons that should remain easy to recover across chats.

## Existing lessons still in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Current-truth pointer docs (`CURRENT_*`) materially reduce restart friction.
- Exact local validation commands, expected outputs, and the next recommended prompt create a tighter and more productive loop.
- Interface/control docs and lightweight façade modules should come before larger refactors.

## New testing/reporting lessons
- Archival test data should be append-only and treated as durable truth.
- Human-readable current reports should be replaceable/regenerable artifacts derived from archival data.
- Apples-to-apples comparisons must use an explicit compatibility key rather than informal “looks similar” judgment.
- Raw performance trends and complexity-normalized projections should be tracked separately.
- Forecasts should be clearly assumption-bound and never presented as high-precision promises.
- The same integrated harness should eventually support smoke, acceptance, regression, benchmark, simulation-credibility, and AI/RL-facing suites through a shared schema.
