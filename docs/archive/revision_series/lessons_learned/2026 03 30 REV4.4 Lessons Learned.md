# 2026 03 30 REV4.4 Lessons Learned

## Purpose
Consolidate durable architecture, workflow, testing, and adapter-boundary lessons.

## Existing validated project lessons that remain in force
- The minimal env can be moved toward the packet boundary without committing the project to a public RL API.
- Deterministic fallback-first replay validation is useful as a low-risk boundary-hardening step before broader RL work.
- Input-stage packet application deserves explicit timing surfacing if thin adapters are expected to be trustworthy.
- Replay capture can start as in-memory full-array snapshots; compression and broader replay-file standards can remain deferred.

## New workflow/process lessons adopted in this pass
### Lesson
The recent failure pattern was **not** lack of lesson capture.
The project already captured many useful lessons.
The stronger pattern was **strong capture with inconsistent follow-through**.

### Best practice now adopted
Narrative lessons must be backed by a thin implementation layer:
- a current lesson-action register
- explicit pre-pass retrieval
- flash-capture during the pass
- end-of-pass synthesis
- concrete destination assignment
- evidence before promotion to best practice

### Best practice now adopted
Every meaningful implementation, testing, relay, or retrospective pass should begin by checking:
- which truth tier is authoritative for the pass
- which open lesson actions are relevant
- which recent lessons are most likely to prevent repeat confusion

### Best practice now adopted
Technical lessons are not "implemented" merely because they were written down.
They become durable best practices only after a successful application with evidence.

### Best practice now adopted
Keep workflow hardening lightweight and markdown-first.
Do not overbuild project-management machinery at this stage.

## Open risks now explicitly recorded
- snapshot-selector behavior remains brittle and still deserves a dedicated compatibility pass
- stale entry points (for example root README and older workflow-facing surfaces) can still mislead future passes if left unattended
- report-path policy and generated-vs-durable artifact boundaries still need a bounded cleanup pass
- if current-pointer freshness and full+sparse reconciliation are treated as optional, repo-truth drift can recur

## Defer rules reinforced
- do not build a heavy project-management subsystem for lessons/actions yet
- do not expand this docs-first process pass into snapshot-script implementation work
- do not let supporting local evidence override repo-truth unless it is explicitly promoted

## POWE promotion candidates strengthened by this pass
- truth-tier discipline: repo-truth vs snapshot-truth vs supporting local evidence
- lesson-to-best-practice promotion only after validation
- pre-pass retrieval of relevant lessons before implementation/testing/relay work
- thin action register on top of narrative retrospective docs
- mandatory freshness/reconciliation audits before authoritative relay/full snapshot publication
