# 2026 03 30 REV3.1 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, and prompting lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Revisioned markdown files are enough for early tracking; do not overbuild project-management tooling yet.
- Root-level install, snapshot, and GitHub scripts reduce restart friction.
- Baseline the project on **Python 3.12.x** for now to reduce package friction across PyBullet, ModernGL, Gymnasium, Numba, and Torch.
- A single `.venv` is simpler than split environments at this stage of the project.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Separating render-subset selection from full simulation state helps identify render-facing costs and keeps stage boundaries cleaner.
- AI scheduling should start as a minimal spec-driven planning layer before introducing real model execution complexity.
- Current-truth pointer docs (`CURRENT_*`) materially reduce restart friction.

## Packet-boundary and ownership lessons already in force
- Explicit `no_change` is meaningfully different from packet reuse.
- Cache-key identity and ownership-lane identity are different concerns.
- Delete and delta packets benefit from clearer baseline and supersede references.
- Not every ownership refresh needs a visible invalidate packet.

## Lifecycle-management lessons now in force
### Benchmark truth is not the whole project truth
For this project, progress should be tracked through at least four coordinated lenses:
- lifecycle status
- verification status
- simulation credibility status
- performance/benchmark status

### Phase gates reduce wasted widening
A new implementation branch is higher leverage when the project explicitly states:
- what phase it is in
- what gate it is trying to clear
- what evidence counts for that gate
- what remains deferred until the gate is cleared

## Verification and simulation-credibility lessons now in force
### Verification and validation must be recorded separately from performance
A fast or well-instrumented subsystem is not automatically validated for the project’s intended use. Verification and validation should be documented alongside benchmark status, not inferred from it.

### Simulation credibility should remain scoped
At the current phase, pyAIGameEngine should make controlled internal prototype claims, not broad physical-truth claims. Intended-use scope should be stated explicitly whenever simulation behavior is discussed.

## AI/RL integration lessons now in force
### RL should remain downstream of engine credibility
Do not let RL convenience reshape the core engine prematurely. Stabilize runtime, packet, and scenario interfaces first, then add environment adapters and learning loops.

### Train/eval separation should be planned before broad RL work
Even before a large RL branch exists, the docs should already assume:
- separate training vs evaluation configurations
- versioned seeds/configs/artifacts
- simple-to-complex scenario progression

## Artifact/versioning lessons now in force
### Non-code artifacts must be treated as first-class project items
Scenario seeds, benchmark configurations, model artifacts, hyperparameters, and candidate snapshots should be tracked as artifacts, not just treated as incidental parameters or filenames.

## Current takeaway
The project now has a stronger management and credibility layer. The next highest-leverage move is to use those new governance docs to drive either a clean `project relay` or a tightly phase-gated next implementation block.
