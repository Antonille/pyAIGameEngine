# 2026 03 30 REV2.3 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, and prompting lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Revisioned markdown files are enough for early tracking; do not overbuild project-management tooling yet.
- Root-level install, snapshot, and GitHub scripts reduce restart friction.
- Baseline the project on **Python 3.12.x** for now to reduce package friction across PyBullet, ModernGL, Gymnasium, Numba, and Torch.
- A single `.venv` is simpler than split environments at this stage of the project.
- Install scripts must create the venv with the intended interpreter, bootstrap `pip` if needed, and verify imports at the end.
- Viewer and benchmark scripts should be runnable independently.
- PyBullet may fall back to source build even on Python 3.12; success depends on a correctly initialized Visual Studio C++ toolchain and Windows SDK environment.
- The install script must never claim success when a package install fails; it must fail fast or explicitly skip optional native packages with a clear warning.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Separating render-subset selection from full simulation state helps identify render-facing costs and keeps stage boundaries cleaner.
- AI scheduling should start as a minimal spec-driven planning layer before introducing real model execution complexity.
- Full snapshots can drift behind sparse implementation passes; current-state reconciliation should explicitly merge sparse artifacts before publishing a new full snapshot.
- Current-truth pointer docs (`CURRENT_*`) materially reduce restart friction.

## 2026-03-29 retrospective additions
### Pass-typing lesson
Prompts are more effective when the pass type is explicit. The most useful pass categories so far are:
- research
- research sweep
- architecture/spec
- implementation
- documentation/interface
- relay/handoff
- retrospective

### Truth-anchor plus non-overcommit lesson
The most reliable prompts do two things together:
- anchor to current validated truth
- explicitly protect against premature overcommitment

Examples of useful guardrails:
- preserve fallback-first architecture
- keep PyBullet optional/deferred
- do not finalize a larger refactor until the interface or runtime-schema boundary is validated

### Validation-loop lesson
Exact local validation commands, expected outputs, and the next recommended prompt create a tighter and more productive loop:
1. change code/docs
2. run exact validation
3. inspect output
4. update conclusions
5. refresh docs
6. issue the next prompt

### Integrated-runtime interpretation lesson
Integrated benchmark/runtime truth should outweigh isolated microbenchmark wins. Warmup state and stage-boundary context must be captured when performance conclusions are recorded.

### Interface-timing lesson
For moving boundaries, interface/control docs and lightweight façade modules should come before a larger refactor. This reduces churn and gives model/AI/UI work a clearer contract target.

### Authoritative-vs-interface representation lesson
Simulation-authoritative structures should not automatically become public interface structures. Model, AI, and UI developers should consume purpose-built contract-facing representations rather than raw internal solver layout.

### Defer-list lesson
“What not to build yet” should be recorded alongside “what to build now.” Explicit defer lists improved decisions in this chat family, especially for:
- full PyBullet dependency
- premature repo/package splits
- overbuilt GPU infrastructure
- early over-generalization of field/shape systems

### Prompt-artifact lesson
Optimized prompts are part of the project operating system, not just convenience text. They should be updated when workflow discipline or best-practice prompts improve.

### Soft-trigger relay lesson
Relay suggestions are better as soft triggers than forced interruptions. Suggest the relay at meaningful milestones and let the user choose whether to relay immediately or complete the current milestone first.

## 2026-03-29 implementation-block additions
### Cadence-accounting lesson
Once model-family cadence is enforced, scheduler runtime bookkeeping must be updated as part of the implemented loop. Planning modules exposed that leaving `last_run_cycle` untouched silently degrades the cadence contract.

### Benchmark-summary lesson
When AI scheduling is cadence-aware, last-step-only benchmark summaries can be misleading. Benchmark tooling should report cumulative schedule counts, transfer batches, entities staged, and bytes staged across the run.

### Transfer-boundary lesson
It is acceptable for the authoritative SoA runtime to retain field-appropriate dtypes while the transfer boundary normalizes staged feature blocks to a model-friendly dtype such as `float32`. The schema/contract layer should make that normalization explicit.

### Registry-before-refactor lesson
A lightweight schema/feature-pack/transfer-planner registry gives enough structure to validate the boundary before a larger refactor. This matched the project rule to prefer contract work before heavy reorganization.

## 2026-03-29 dual-family bridge additions
### Output-contract lesson
A second scheduled family is a better validation of the boundary when it changes not only cadence and feature width but also the **output contract**.

### Packet-versioning lesson
Even a lightweight action bridge benefits from explicit packet versioning, cycle metadata, source-family labeling, and deadline metadata. These are cheap to add early and expensive to retrofit later.

### Replace-vs-delta lesson
Direct replacement actions and contextual delta overlays are worth separating early. They create cleaner expectations for application semantics and make later caching/baseline work easier.

### Workload-mix lesson
Integrated benchmark winners can flip when the workload mix changes even if one backend remains stronger in a specific stage such as physics. Benchmark claims must stay tied to the tested workload.

## 2026-03-30 research-sweep additions
### SoA-still-right-now lesson
The research sweep reinforces the current direction: **SoA is still the correct near-term default** for the hot path. Data-oriented ECS references emphasize contiguous field storage, batching, and cache/vectorization benefits, which matches the current runtime-schema direction.

### AoSoA-trigger lesson
AoSoA is a plausible **later** path, not a current rewrite target. It becomes more interesting when field count grows, random access patterns worsen, or vectorization/cache pressure suggests a pure SoA layout is no longer the best tradeoff.

### Scheduler-mode lesson
AI family scheduling should be treated more explicitly as a **batching-strategy space**, not just a cadence knob. Continuous, chunked, mixed, and disaggregated-like scheduling modes are worth comparing once the current packet boundary is stable.

### Packet-baseline lesson
Delta-style action packets need explicit baseline semantics. Add generation or baseline identifiers, and explicit meanings for “unchanged,” “removed,” or “invalidated,” before making packet caching more sophisticated.

### Tool-gating lesson
Keep **NumPy + Numba** as the working default. Consider **Pythran** or **Cython** only for stable numeric kernels that stop benefiting from Numba. Consider **ONNX Runtime** only after model-family serving contracts stabilize. Consider **CuPy** only after the CPU-authoritative runtime/schema/packet boundary is stable enough that GPU migration targets are clear.

### Benchmark-matrix lesson
When benchmark winners are mixed, the answer is not a snap rewrite. The correct next move is a **benchmark matrix** that varies run length, body count, family count, cadence mix, feature width, and backend.

### Research-sweep procedure lesson
When benchmark truth is mixed or the architecture branch is widening, use a named **research sweep** before further implementation. This keeps the project aligned with outside best practices without forcing premature refactors.

## Candidates to promote into the Prompt Optimization and Workflow Engineering knowledge base
Promote these as broadly reusable, not just project-local:
- explicit prompt structure: truth anchor + pass type + constraints/deferrals + deliverables + validation + next-step handoff
- pass-clean prompting instead of mixed-purpose prompting
- milestone-based soft relay triggering
- exact validation loop discipline
- interface-before-refactor timing
- integrated-runtime truth over isolated microbenchmark wins
- explicit defer-list / not-yet guidance in prompts and specs
- research sweep before widening an architecture branch with ambiguous benchmark results
- benchmark matrix framing instead of single benchmark cherry-picking

## Current takeaway
The research sweep supports the current architecture direction. It does **not** justify abandoning fallback-first CPU-authoritative SoA work. It does justify making the next step more specification-driven around packet-cache semantics, baseline/generation semantics, and benchmark-matrix design before another larger implementation jump.
