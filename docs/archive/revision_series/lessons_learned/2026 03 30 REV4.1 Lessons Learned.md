# 2026 03 30 REV4.1 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, prompting, packaging, relay, and repo-discipline lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Baseline the project on **Python 3.12.x** for now to reduce package friction.
- A single `.venv` is simpler than split environments at this stage.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Integrated benchmark/runtime truth should outweigh isolated microbenchmark wins when representative evidence is available.
- Smoke runs are for instrumentation visibility; representative runs should drive benchmark claims.

## Repo and workflow additions from this pass
### GitHub-primary collaboration lesson
- Once the repo exists and the intended baseline is pushed, GitHub should become the primary shared workflow surface.
- Snapshots remain valuable, but they should not quietly replace repo discipline.

### Transition-state lesson
- A new or empty repo does **not** automatically supersede a newer local tree or uploaded candidate snapshot.
- During transition, the immediate truth may still be local or snapshot-based until the repo catches up.

### Repo-creation capability lesson
- Do not assume the connector can create repos on demand.
- Repo creation may need to be done by the user, after which the workflow can switch to GitHub-first.

### Snapshot-secondary lesson
- Snapshot exports are best treated as archive, relay, rollback, and recovery artifacts once GitHub is active.
- Snapshot metadata should reference repo identity and commit/branch when available.

## Relay and truth-establishment lessons still in force
- When a candidate snapshot is uploaded, use **Python first** to inspect it.
- Every full and sparse snapshot should include a root-level `OPEN_THIS_FIRST.md` and `SNAPSHOT_INDEX.json`.
- Relay reconciliation must check that docs do not claim code truth absent from the candidate snapshot.

## Prompt and workflow takeaways
- Prompts are more effective when the pass type is explicit.
- The most reliable prompts combine:
  - current-truth anchor
  - constraints/deferrals
  - exact deliverables
  - exact validation commands
  - next-step handoff
- Relay suggestions are better as soft triggers than forced interruptions.
- Repo sync state should now be called out explicitly in handoffs when relevant.
