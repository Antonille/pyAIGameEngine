# 2026 03 30 REV3.8 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, prompting, and governance lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Revisioned markdown files are enough for early tracking; do not overbuild project-management tooling yet.
- Root-level install, snapshot, and GitHub scripts reduce restart friction.
- Baseline the project on **Python 3.12.x** for now to reduce package friction across PyBullet, ModernGL, Gymnasium, Numba, and Torch.
- A single `.venv` is simpler than split environments at this stage of the project.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Integrated benchmark/runtime truth should outweigh isolated microbenchmark wins when current evidence is available.
- Smoke runs are for instrumentation visibility; representative runs should drive the current winner claim.
- Apples-to-apples benchmark history needs explicit compatibility keys and stable benchmark subsets.

## Governance and workflow lessons now in force
- Lifecycle status, verification status, simulation credibility status, and performance status should be kept as separate truth sources.
- RL/AI integration should remain downstream of engine credibility and stable packet/interface boundaries.
- Gate-clearing passes are higher leverage when they produce explicit automated acceptance checks rather than only prose conclusions.
- Research sweeps are more useful when they include at least one complementary tooling/workflow/testing topic beyond the main architecture survey.

## Complementary research additions in REV3.8
### Property-based/stateful testing lesson
Property-based and stateful tests are a strong complement to gate-oriented acceptance scripts for subsystems whose behavior is naturally described by invariants:
- scheduler cadence/freshness/fallback/batchability
- packet-lane generation/baseline/supersede behavior
- adapter mode/seed metadata behavior
- append-only archival schema behavior

### Benchmark-history lesson
ASV is a strong reference model for long-lifecycle benchmark history even if the project does not adopt it immediately. The current custom harness should preserve compatibility concepts now:
- machine/environment metadata
- benchmark compatibility groups
- stable subsets for apples-to-apples comparison
- on-disk structured result history

### Experiment-tracking lesson
DVC and MLflow are useful future candidates, but they are not required now. The project should first build a lightweight append-only archival truth model and only adopt heavier tracking systems when experiment scale justifies them.

### Adapter-guidance lesson
Official environment guidance reinforces that adapter-facing paths should keep:
- explicit seeding
- explicit mode tagging
- observation + info style reset semantics
without turning the adapter into the authoritative simulation path.

## Current takeaway
The current architecture path remains well-supported. The highest-value improvements right now are not architecture reversals; they are stronger long-lifecycle testing/reporting design, invariant-oriented validation, and future-friendly benchmark-history discipline.
