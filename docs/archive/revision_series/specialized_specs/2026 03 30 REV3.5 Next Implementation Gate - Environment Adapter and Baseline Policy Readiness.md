# 2026 03 30 REV3.5 Next Implementation Gate - Environment Adapter and Baseline Policy Readiness

## Purpose
This document names the next recommended implementation gate after clearing Phase 3 Gate D.

## Recommended next gate
**Phase 4 Gate A — Environment Adapter and Baseline Policy Readiness**

## Why this gate comes next
This gate should come before broader RL implementation because:
- AI scheduling and packet acceptance now have explicit acceptance checks
- lifecycle / V&V / simulation-credibility governance is in place
- the next highest-leverage step is to define the engine-to-experiment boundary and a minimal baseline-policy path
- this preserves the project rule that RL remains downstream of engine credibility, not the immediate driver

## What the gate should require
- an architecture/spec for a Gymnasium-compatible adapter boundary
- explicit train/eval separation rules for early experiments
- at least one scripted or heuristic baseline policy plan
- artifact/versioning expectations for seeds, configs, and policy/model outputs
- explicit acceptance criteria for the first adapter-facing validation scenario

## What should remain deferred
- broad RL algorithm support
- large experiment matrices
- GPU-authoritative runtime decisions
- broader model-family complexity unrelated to the adapter boundary

## Recommended next pass type
**Architecture/spec pass**

## Desired output of that pass
- recommended adapter boundary
- baseline-policy scope
- acceptance criteria
- validation commands / evidence plan
- next implementation prompt
