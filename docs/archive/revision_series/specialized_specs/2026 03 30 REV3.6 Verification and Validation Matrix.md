## Purpose
This matrix defines what pyAIGameEngine currently claims, how those claims are checked, and what evidence is expected before the project treats a subsystem as accepted for the current phase.

- engine/runtime verification
- packet/action boundary verification
- benchmark tooling verification
- simulation behavior verification
- AI scheduling verification
- environment-adapter readiness verification

## Phase 4 Gate A row

### Claim
The project is ready to expose a thin, controlled, AI/RL-facing adapter boundary without destabilizing the current engine-side truth.

### Acceptance checks required
- adapter exists and is non-authoritative
- adapter mode tagging exists for `train`, `eval`, `benchmark`
- adapter reports:
  - `scenario_version`
  - `adapter_version`
  - `seed`
  - `baseline_policy_id`
- hold baseline works
- simple reflex baseline works
- current smoke benchmark and rigid-body/field validation still pass after adapter introduction

### Required evidence
- Gate A acceptance script output
- one hold rollout output
- one simple reflex rollout output
- re-run of smoke benchmark
- re-run of rigid-body / field validation

## Current recommendation
Use this matrix to decide whether the next implementation pass has cleared Gate A explicitly, rather than inferring readiness from ad hoc rollout behavior.
