# 2026 03 30 REV3.7 Archival Test Data Schema

## Purpose
This document defines the append-only archival schema for pyAIGameEngine test history.

## Primary durable artifact
Recommended primary file:
- `artifacts/test/archive/test_run_archive.jsonl`

Recommended optional future mirrors:
- `artifacts/test/archive/test_run_archive.parquet`
- `artifacts/test/archive/index/compatibility_index.json`

## Append-only rule
Each line in `test_run_archive.jsonl` represents one completed test run record. Existing records are never mutated. Any correction should be recorded as a new run record with a superseding note.

## Top-level run record fields
- `schema_version`
- `test_harness_version`
- `run_id`
- `timestamp_utc`
- `snapshot_id`
- `workspace_version`
- `poc_version`
- `project_phase`
- `gate_context`
- `execution_mode` (`train`, `eval`, `benchmark`, `mixed`)
- `host_metadata`
- `environment_metadata`
- `git_or_snapshot_notes`
- `suite_selection`
- `parameter_bundle`
- `validation_summary`
- `performance_summary`
- `comparison_summary`
- `projection_summary`
- `development_state_summary`
- `design_state_summary`
- `artifact_references`
- `suite_results`
- `notes`

## Suite result fields
Each entry in `suite_results` should contain:
- `suite_id`
- `suite_category`
- `suite_version`
- `status`
- `checks_passed`
- `checks_failed`
- `checks_skipped`
- `metrics`
- `parameters_used`
- `artifacts`
- `summary`

## Metric record fields
- `metric_name`
- `metric_value`
- `metric_unit`
- `metric_kind` (`validation`, `performance`, `count`, `derived`, `projection`)
- `comparison_key`
- `notes`

## Development state summary fields
- `implemented_components`
- `unimplemented_components`
- `current_gate_status`
- `current_limitations`
- `assumptions`

## Design state summary fields
- `adopted_design_choices`
- `deferred_design_choices`
- `currently_evaluated_choices`

## Artifact reference fields
- `artifact_type`
- `relative_path`
- `description`
- `content_hash` optional
- `producer`

## Compatibility keys for comparison
The archival schema should explicitly store comparison keys for apples-to-apples grouping:
- `benchmark_family_id`
- `scenario_bundle_id`
- `adapter_version`
- `baseline_policy_id`
- `backend_mode`
- `steps`
- `bodies`
- `feature_family_set`
- `packet_mode_set`
- `mode`
- `seed_policy`

## Notes
This schema is intentionally verbose early in project life. Compression and denormalization are acceptable later, but explicitness is more valuable now.
