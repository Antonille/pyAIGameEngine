# 2026 03 30 REV3.7 Testing and Reporting Workflow

## Purpose
This document defines the intended long-cycle local workflow for pyAIGameEngine testing and reporting.

## Intended local workflow
1. the assistant provides test-harness calling syntax
2. the user runs the test harness locally
3. the harness appends a new archival record and emits current report artifacts
4. the user uploads the archival data file and any generated images/artifacts when needed
5. the assistant validates the run results
6. accepted results are integrated into the next snapshot
7. the assistant regenerates or refines the current human-readable report as needed

## Local command philosophy
The harness should support:
- one-command default integrated run
- targeted suite selection when needed
- explicit benchmark parameters when needed
- minimal required prompting at the terminal

## Recommended default command shape
- integrated all-up current-state run
- integrated benchmark-only run
- report regeneration-only run
- archive append validation-only run

## Failure handling
If a run is invalid:
- do not treat it as authoritative benchmark truth
- preserve the raw record if useful, but mark it invalid or superseded
- regenerate the current report only from accepted runs

## Snapshot integration rule
Only accepted test results should be integrated into the next truth-setting snapshot.
