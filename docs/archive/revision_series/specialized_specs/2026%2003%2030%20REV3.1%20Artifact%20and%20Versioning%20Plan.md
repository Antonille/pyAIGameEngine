# 2026 03 30 REV3.1 Artifact and Versioning Plan

## Purpose
This document defines a lightweight artifact/versioning discipline for pyAIGameEngine so code, scenarios, benchmark configs, model artifacts, and candidate snapshots can be compared and reproduced more reliably.

## Artifact classes

### 1. Candidate snapshots
Examples:
- sparse candidate snapshot
- full candidate snapshot

Rules:
- snapshots remain the main handoff vehicle
- sparse snapshots capture only changed project files
- full snapshots capture the candidate full project state
- naming can continue to be managed centrally in-chat unless the user explicitly changes that rule

### 2. Scenario seeds and scenario definitions
Examples:
- deterministic test seeds
- benchmark scenario seeds
- reset-state definitions

Rules:
- treat seeds/configs as artifacts, not incidental parameters
- keep scenario identifiers stable when possible
- record changed seeds/configs when benchmark meaning changes

### 3. Benchmark configurations
Examples:
- steps
- body count
- backend mode
- warmup settings
- family mix / cadence mix when applicable

Rules:
- benchmark claims should always name the configuration class or exact command
- representative subsets should be distinguished from smoke runs

### 4. Model artifacts
Examples:
- weights/checkpoints
- exported inference artifacts
- feature-pack versions
- packet-schema compatibility notes

Rules:
- do not treat model artifacts as interchangeable with code revisions
- record which packet/action contract version a model artifact expects

### 5. Hyperparameters and experiment configs
Examples:
- training hyperparameters
- evaluation configs
- reward shaping settings
- rollout horizon settings

Rules:
- treat experiment configs as versioned inputs to the result, not just notes

## Lightweight naming guidance
Recommended pattern:
- code/doc candidate: existing snapshot naming pattern
- scenario artifact: `scenario_<name>_v<major.minor>.json` or similar later
- benchmark config: `benchmark_<class>_v<major.minor>.json` or similar later
- model artifact: `model_<family>_<contract>_v<major.minor>` later when needed

## Versioning posture now
Current practical version layers:
- workspace version
- POC package version
- revisioned docs
- candidate snapshots
- packet/action protocol versions

This is sufficient for the current phase if the artifact classes above are tracked intentionally.

## Baseline rules
- A benchmark claim should reference the relevant code/doc baseline and the benchmark configuration.
- A simulation-behavior claim should reference the scenario or seed class when relevant.
- A future RL result should reference model artifact version, environment/API version, and training/eval config versions.

## Current recommendation
Use this plan lightly but consistently:
- do not build a heavy artifact-management system yet
- do start naming and recording non-code artifacts as first-class project items
- make candidate snapshots the main integration and relay baseline until the project reaches a larger collaboration milestone
