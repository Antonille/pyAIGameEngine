# 2026 03 30 REV2.4 Packet Cache and Packet-State Modes at the Action Boundary

## Purpose
This document records the packet-cache and packet-state implementation added on top of the existing runtime-schema / transfer-planner / action-packet bridge.

## What was added
- explicit `generation_id` and `baseline_generation_id` fields on `AIActionPacket`
- packet cache keyed by:
  - source family
  - actor runtime family
  - action-block schema
  - entity-batch signature
- cache-outcome logging at the packet boundary
- support in the action layer for packet states:
  - `replace`
  - `delta`
  - `invalidate`
  - `no_change`
  - `delete`
- packet-layer benchmark metrics and summaries
- additional output contract: `action4_delete_v1`
- additional validating family: `agent_cleanup_v1`

## Current semantics
### Cache hit
If the packet cache key matches, the staged feature signature matches, and the cached generation is still within freshness limits, the bridge reuses the cached packet rather than rebuilding a new payload.

### Cache miss
If no cache entry exists, the bridge builds a new packet generation.

### Cache invalidation
If a cache entry exists but cannot be reused, the bridge records an invalidation and emits an `invalidate` packet before building the new packet generation.

### Delete contract
`action4_delete_v1` zeros owned action rows for flagged actors. This exercises a packet contract that must often be reapplied on reuse rather than replaced with a no-op marker.

## Why this matters
This strengthens the packet boundary without overcommitting to a broader runtime refactor. The packet cache remains contract-facing, benchmark-visible, and aligned with the project rule that interface boundaries should be clarified before larger architecture moves.

## Remaining open questions
- whether to add an explicit `no_change` output contract, not just support for the mode
- whether delete/invalidate ownership should later become more granular than the current block-level approach
- what benchmark thresholds should justify future AoSoA or alternate tool experiments
