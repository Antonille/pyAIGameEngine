# 2026 03 30 REV3.1 Simulation Credibility Plan

## Purpose
This document distinguishes performance and software progress from simulation credibility. It defines how pyAIGameEngine should reason about the credibility of simulation behavior during its current lifecycle stage.

## Current intended-use scope
At the present stage, the simulation layer is intended for:
- architecture exploration
- data-layout and interface experimentation
- packet/action-boundary testing
- controlled benchmark and prototype scenario work

It is **not yet intended** to support broad physical-truth claims, scientific publication claims, or production-quality reinforcement-learning environment claims.

## Credibility dimensions

### 1. Software verification
Question:
Does the implemented code behave as designed?

Current examples:
- fallback path validation
- runtime-schema / transfer-planner behavior
- packet boundary mode handling
- benchmark tooling consistency

### 2. Model / simulation verification
Question:
Does the simulation model behave internally the way the chosen model says it should?

Current examples:
- rigid-body / gravity / field prototype checks
- packet-action application consistency for the current model assumptions

### 3. Validation
Question:
Is the simulation behavior acceptable for the intended use?

Current posture:
- acceptable for internal architecture exploration and controlled prototype work
- not yet validated for broad general-purpose simulation claims

### 4. Uncertainty / sensitivity
Question:
How much do results depend on parameters, scenario setup, or modeling assumptions?

Current posture:
- recognized but still lightly documented
- should be expanded before simulation claims widen

### 5. Intended-use scope control
Question:
Are users of the project likely to misread benchmark or prototype results as stronger simulation claims than intended?

Current requirement:
- every major simulation-related doc pass should state current intended-use scope clearly

## Credibility levels for this project

### Level C0 — exploratory prototype credibility
Suitable for:
- architecture reasoning
- internal prototyping
- interface experiments
- workflow and performance decisions

### Level C1 — controlled scenario credibility
Suitable for:
- repeated scenario checks with explicit assumptions
- limited internal comparison studies
- baseline environment behavior for early AI experiments

### Level C2 — broadened scenario credibility
Not yet reached.
Would require:
- broader scenario coverage
- stronger validation targets
- uncertainty/sensitivity notes
- better traceability between model assumptions and observed behavior

## Current project position
pyAIGameEngine is currently between **C0 and early C1** depending on subsystem.
The rigid-body/field prototype and packet/action behavior are usable for controlled internal reasoning, but not yet for strong simulation-truth claims.

## What counts as credibility progress next
- more scenario-based checks, not only one current smoke/prototype scenario
- clearer mapping between intended use and accepted evidence
- explicit sensitivity notes for major scenario parameters
- stronger simulation-specific validation records separate from runtime benchmarks

## Guardrails
- Do not let raw benchmark wins stand in for simulation credibility.
- Do not claim validation beyond the intended-use scope recorded here.
- Keep simulation credibility status separate from code maturity and from backend winner claims.

## Immediate recommendation
Use the Verification and Validation Matrix together with this document whenever a new simulation-facing subsystem or scenario is added.
