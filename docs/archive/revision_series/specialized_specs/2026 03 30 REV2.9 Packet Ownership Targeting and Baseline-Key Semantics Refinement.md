# 2026 03 30 REV2.9 Packet Ownership Targeting and Baseline-Key Semantics Refinement

## Purpose
This document records the implementation pass that refined packet ownership and baseline semantics after the explicit `no_change` workload had a validated benchmark subset.

## What changed
- Extended `AIActionPacket` with:
  - `baseline_ownership_key`
  - `supersedes_ownership_key`
- Advanced the packet protocol version to **`action_packet_v4`**.
- Kept `ownership_scope` and `ownership_key`, but clarified that packets may also need to point at:
  - the ownership key used as the semantic baseline
  - the ownership key being retired or superseded
- Refined packet metadata to include clearer `ownership_transition` descriptions.
- Improved summary/log output so packet lines now show:
  - baseline generation and baseline ownership key
  - superseded generation and superseded ownership key
  - current ownership scope and current ownership key

## Semantic intent of the refinement
### Replace
- A replace packet may install a fresh concrete ownership key.
- If it replaces an active owner, it now carries clearer supersede references to the earlier owned lane state.

### Delta
- A delta packet does not necessarily require a separate visible invalidate packet.
- It should still identify the prior lane state it is baselining from when one exists.

### Delete
- A delete packet can now identify the owned lane state it is retiring more clearly.
- Delete semantics do not need an extra visible invalidate packet when the delete packet already identifies the target lane lineage clearly enough.

### Invalidate
- Invalidate remains available when a scope transition or explicit ownership-lane rebinding needs a visible invalidation event.
- It now carries both generation-based and ownership-key-based references to the retired lane state.

- Packet lines show clearer lineage data through baseline and superseded ownership keys.
- In the local smoke workload, packet counts dropped relative to the explicit-invalidate-heavy version because ownership-target references reduced the need for extra invalidate packets.
- This pass intentionally did **not** replace the last validated benchmark truth in the current-truth docs. The next benchmark-compare pass should evaluate this ownership-aware workload explicitly.
