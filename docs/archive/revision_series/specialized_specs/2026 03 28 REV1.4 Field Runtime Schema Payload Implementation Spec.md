# 2026 03 28 REV1.4 Field Runtime Schema Payload Implementation Spec

## Purpose
This document packages the current field, coordinate-system, runtime-schema, and property/payload architecture into a repo-ready implementation spec. It extends the active architecture direction with a concrete **runtime schema + field payload prototype** and gives modules, class sketches, and milestone steps.

## Current truth assumptions
- Workspace version: **0.0.9**
- Active proof of concept: **POC1_SoaFirst**
- Rendering remains **2D for now**, but engine math/state/physics stay **3D**
- Physics-bearing coordinate systems are **rigid and inertial only**
- Runtime hot-path design favors **SoA layouts**
- Fallback-first execution remains authoritative; PyBullet is optional/deferred

## A. Architecture decisions in force
1. The master stationary reference frame owns the authoritative universal gravity vector and shared physical constants.
2. Embedded coordinate systems inherit and transform gravity/field context from parent coordinate systems.
3. Runtime gravity is stored as a **3-vector acceleration**, not as unit-direction + signed-magnitude in the hot path.
4. Static and dynamic field sources must be architecturally distinct.
5. Repeated object families should use **fixed runtime schemas + optional sparse property payloads**.
6. Complex geometry/property descriptions should be layered behind handles, masks, and payload stores rather than forcing huge inline runtime records.

## B. Runtime schema + field payload prototype
### B1. CoordinateSystemRuntimeSchema
Suggested SoA fields:
- `coord_system_id: int32`
- `parent_coord_system_id: int32`
- `flags: uint32`
- `transform_revision: uint32`
- `inherited_field_revision: uint32`
- `parent_field_revision_seen: uint32`
- `parent_transform_seen: uint32`
- `pos_parent_xyz: float32[3]`
- `rot_parent_quat: float32[4]`
- `R_parent_to_local: float32[3,3]`
- `R_local_to_parent: float32[3,3]`
- `gravity_local_xyz: float32[3]`
- `G_like_local: float32`
- `epsilon0_like_local: float32`
- `mu0_like_local: float32`
- `static_field_atlas_id: int32`
- `dynamic_partition_id: int32`

### B2. GravityPropagationManager
Responsibilities:
- own master gravity revision
- walk coordinate-system graph top-down
- update cached local gravity/constant context when parent inherited field revision, local transform revision, or parent constants revision changes

Pseudo-rule:
```text
if parent_field_revision_seen != parent.inherited_field_revision
or parent_transform_seen != transform_revision
or flags & CS_FIELD_DIRTY:
    gravity_local = R_parent_to_local * gravity_parent
```

### B3. FieldSourceSchema
Each source-capable object exposes:
- `field_source_id`
- `owner_entity_id`
- `coord_system_id`
- `field_mask`
- `source_mode` (`none`, `static`, `dynamic`, `area_effect`)
- `cutoff_mode` (`none`, `radius`, `threshold_derived`)
- `cutoff_radius`
- `cutoff_radius_sq`
- `threshold_value`
- `gravity_like_strength`
- `electric_like_strength`
- `magnetic_like_vec: float32[3]`
- `area_effect_kernel_id`
- `payload_id`

### B6. Sparse payload stores
Only objects that need extra complexity receive payload entries.
Prototype stores:
- `ChargePayloadStore`
- `MagneticPayloadStore`
- `AreaEffectPayloadStore`
- `HeterogeneousPropertyStore`
- `VoxelChunkStore`

The runtime header remains small and SoA-friendly; the payload stores hold rare or detailed data.

### B7. StaticFieldAtlas
Recommended early static-field representation:
- regular sampled grid per coordinate-system region
- vector-field samples for gravity/electric/magnetic-like channels
- trilinear interpolation at runtime

Suggested fields:
- `atlas_id`
- `coord_system_id`
- `bounds_min_xyz`
- `bounds_max_xyz`
- `resolution_xyz`
- `cell_size_xyz`
- `gravity_like_samples[...,3]`
- `electric_like_samples[...,3]`
- `magnetic_like_samples[...,3]`
- `atlas_revision`
- `source_revision_seen`

### B8. DynamicFieldSolver
Recommended early strategy:
- uniform grid or spatial hash
- cutoff-limited neighbor evaluation
- vectorized / JIT-eligible accumulation over candidate neighbors

Do **not** start with all-pairs caching of per-target contributions.

### B9. AreaEffectSchema
Treat area effects as special field sources with:
- origin / frame
- support radius or support volume
- falloff kernel
- lifetime window
- effect channel mask

## C. Geometry/property architecture
Use layered geometry modes:
- `primitive`
- `compound`
- `voxel_chunk`
- defer `sampled_sdf` / hybrid until later

Use layered property modes:
- uniform material defaults
- shape-average properties
- heterogeneous/property-varying payloads

Interpretation of project vocabulary:
- **voxel** = geometry/boundary occupancy element
- **voxellino** = scalar-valued volumetric/property sample

Do **not** force voxel and voxellino to be identical runtime structures.

## E. Class sketches
### `CoordinateSystemStore`
Responsibilities:
- hold coordinate-system SoA buffers
- parent/child relationships
- transform revisions
- inherited field revisions
- local gravity cache

### `GravityPropagationManager`
Responsibilities:
- own/update master gravity revision
- propagate local inherited gravity/constants
- expose `propagate_dirty_coordinate_systems()`

### `FieldSourceStore`
Responsibilities:
- own `FieldSourceSchema` rows
- classify sources into static/dynamic/area-effect groups
- expose source subsets per coordinate system

### `StaticFieldAtlasManager`
Responsibilities:
- rebuild atlases when static source revisions change
- answer field queries by interpolation

### `DynamicFieldSolver`
Responsibilities:
- query dynamic-neighbor candidates from partition structure
- evaluate cutoff-limited interactions
- accumulate per-target field/force contributions

### `PropertyPayloadRegistry`
Responsibilities:
- centralize payload-handle -> concrete store mapping
- keep optional sparse payload stores discoverable without widening hot-path rows

### `VoxelChunkStore`
Responsibilities:
- store chunked scaled-voxel geometry
- optional heterogeneous property references per chunk or region

### Milestone 4 — dynamic field broad-phase prototype

- validate one shape with heterogeneous or chunked properties
