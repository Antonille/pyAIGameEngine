# 2026 03 29 REV1.5 Rigid-Body, Field, and Runtime-Schema Implementation Spec

## Purpose
This document turns the current architecture direction into a repo-ready implementation spec for the next design/prototype pass. It is written for a reader with an undergraduate physics background and is intended to bridge physical reasoning and engine implementation.

## Scope
This spec covers:
- simulation frames vs body-local frames
- rigid-body runtime schema
- primitive/compound geometry representation
- collision/contact data flow
- center-of-mass field approximation as the first coupling model
- a staged upgrade path to distributed body-force samples
- runtime schema + field payload prototype
- milestone ordering

It does **not** attempt to fully specify:
- exact solver math for every primitive pair
- high-fidelity electromagnetic simulation
- a complete voxelized collision or field engine
- non-rigid or non-inertial global simulation frames

## Architecture decisions now in force
1. Simulation state remains fully **3D**, even while rendering stays **2D** for now.
2. Physics-bearing world/scene coordinate systems remain **rigid-body transforms** only.
3. Not all coordinate systems are inertial:
   - scene/master/ambient simulation frames are treated as inertial for current POC work
   - body-local / shape-local frames are rigidly attached to shapes and are generally **non-inertial**
4. The master stationary reference frame owns universal constants and the master gravity vector.
5. The master gravity vector may vary with time and must propagate efficiently to child simulation coordinate systems.
6. First-pass field coupling treats each rigid body as a **point interaction at its center of mass**.
7. Collision and boundary detection operate on body-defined geometry transformed into a common collision frame.
8. Runtime hot-path storage continues to favor **SoA + fixed runtime schemas + optional sparse payloads**.[^1][^2]

## 1. Frame taxonomy

### 1.1 Simulation coordinate systems
These are the frames used to evaluate forces, propagate ambient fields, integrate rigid-body state, and solve collision impulses.

Properties:
- rigidly related to parent
- inertial in current POC assumptions
- used for:
  - inherited field propagation
  - field queries
  - rigid-body integration
  - collision/contact solve
  - render snapshot generation

### 1.2 Body-local / shape-local frames
These are attached to shapes or rigid bodies and move with them.

Properties:
- rigidly attached to the body
- generally non-inertial because the body may accelerate and rotate
- used for:
  - defining geometry
  - storing local inertia tensors
  - storing local sample points
  - storing local voxel/scaled-voxel geometry
  - storing local heterogeneous property payloads

**Key rule:** body-local frames are description frames, not primary force-balance solve frames.[^3][^4]

## 2. Rigid-body runtime schema

Create `poc1_engine/runtime/rigid_body_store.py`.

### 2.1 Core SoA runtime storage
```text
entity_id                   int32[N]
coord_system_id             int32[N]
flags                       uint32[N]

pos_world_xyz               float32[N,3]
rot_world_quat              float32[N,4]

linvel_world_xyz            float32[N,3]
angvel_world_xyz            float32[N,3]

force_accum_world_xyz       float32[N,3]

inv_mass                    float32[N]

inertia_body_diag           float32[N,3]
inv_inertia_body_diag       float32[N,3]

geometry_mode               uint8[N]

material_id                 int32[N]

field_source_handle         int32[N]
```

### 2.2 Rationale
- `mass`, `inv_mass`, `inertia_body_diag`, and `inv_inertia_body_diag` are hot-path physics quantities.
- Geometry, material, and payload handles keep the common row small.
- This layout supports both rigid-body physics and later AI feature extraction from fixed runtime schemas.[^1][^5]

### 3.2 Geometry query interface
All geometry modes should grow toward a common interface:
```python
class GeometryQueryInterface:
    def bounds_local(self): ...
    def point_inside_local(self, p_local): ...
    def support_local(self, direction_local): ...
    def closest_point_or_distance_local(self, p_local): ...
```

### 3.3 Why this matters
This allows:
- broad phase from transformed bounds
- narrow phase from primitive-pair routines first
- later support for GJK/EPA or voxel-specific queries
- shape-local storage with world/common-frame contact solving[^5][^6]

### 4.2 Narrow phase
Early primitive pairs:
- sphere-sphere
- sphere-box
- box-box

### 4.3 Contact data structure
```python
from dataclasses import dataclass
import numpy as np

@dataclass
class ContactPoint:
    body_a: int
    body_b: int
    world_point: np.ndarray        # (3,)
    world_normal: np.ndarray       # (3,), points from A to B
    penetration_depth: float
    ra_world: np.ndarray           # p - x_a
    rb_world: np.ndarray           # p - x_b
```

### 4.4 Contact manifold
```python
@dataclass
class ContactManifold:
    body_a: int
    body_b: int
    contacts: list[ContactPoint]
```

### 4.5 Solver direction
Use **small contact manifolds + sequential impulses** as the first robust path, not a one-contact-only toy solver.[^7][^8]

## 5. Rigid-body response: why a spinning box differs from a sphere

Two bodies with the same mass can still respond differently because:
1. the contact manifold differs
2. the inertia tensor differs

### 5.1 Inertia storage
Store body-frame inverse inertia and rotate it into world coordinates each step:
```text
I_world^-1 = R * I_body^-1 * R^T
```

### 5.2 Early primitive inertia formulas
For first POC implementation:
- sphere: isotropic inertia
- box: diagonal body-frame inertia with axis-dependent moments
- capsule: analytic or deferred if necessary

### 5.3 Why this is enough early
Primitive collision geometry + correct inertia tensors already gives visibly different spinning-collision behavior for spheres vs boxes, without requiring voxelized collision or volumetric impulse distribution.[^3][^4][^7]

## 6. Field coupling architecture

## 6.1 First-pass coupling rule
Treat each rigid body as a point interaction located at its center of mass.

Evaluate:
- net inherited ambient gravity at the body's simulation-frame position
- net static/dynamic field contributions at the center of mass
- apply resulting force at the center of mass
- no field-gradient torque in first pass unless explicitly modeled

This is the cheapest correct-enough starting point.

### Why it is acceptable
For:
- uniform gravity
- slowly varying fields
- gameplay analog gravity/electric-like fields

the center-of-mass approximation is often sufficient for early rigid-body behavior and preserves a clean upgrade path.[^3]

---

## 6.2 Upgrade path: distributed interaction samples
Later, if you want a field to act differently across a body's volume, add an optional interaction-sample payload.

Create:
- `poc1_engine/fields/interaction_sample_store.py`

### 6.2.1 Sample payload
Each body may have `k` body-local interaction samples:
```text
sample_local_pos_xyz        float32[k,3]
sample_weight               float32[k]
sample_property_mask        uint32[k]
sample_charge_like          float32[k]
sample_mass_fraction        float32[k]
sample_magnetic_moment      float32[k,3]
```

### 6.2.2 Runtime evaluation
For each sample:
1. transform local sample to world/common frame
2. query field at sample point
3. accumulate:
   - net force
   - net torque about center of mass

```text
F_total = Σ F_i
τ_total = Σ r_i × F_i
```

This is the recommended bridge between:
- point approximation
- and fully heterogeneous volumetric coupling

### 6.2.3 Why not go directly to full voxels
Weighted sample points are usually the better middle step:
- cheaper
- easier to debug
- easier to tune
- still captures torque from field variation across the body

## 8. Voxel / voxellino integration path

### 8.1 Recommended interpretation
- **voxel** = geometric/occupancy/surface-support element
- **voxellino** = scalar-valued property sample or volumetric property carrier

### 8.2 Early runtime rule
Do not make every body a dense voxel object.
Instead:
- keep primitives and compounds as the early collision path
- allow a voxel/scaled-voxel payload mode for selected objects
- later derive interaction samples from voxel or property chunks when richer field coupling is needed

### 8.3 Why this matters
This preserves flexibility without making the whole engine memory-heavy or collision-heavy from day one.[^9]

## 9. Milestone steps

## Milestone 1 — frame separation and rigid-body schema
Implement:
- `RigidBodyStore`
- `CoordinateSystemStore`
- explicit distinction between simulation frames and body-local frames
- analytic inertia support for spheres and boxes

Validation:
- spawn spheres and boxes with different inertias
- confirm distinct rotational response

## Milestone 2 — primitive collision/contact path
Implement:
- broad phase AABB
- narrow phase for sphere-sphere and sphere-box
- `ContactPoint`
- `ContactManifold`
- initial sequential-impulse solve

Validation:
- box vs sphere collision cases
- spinning box vs spinning sphere behavior

## Milestone 3 — gravity propagation manager
Implement:
- master gravity vector with revision tracking
- child simulation-frame gravity propagation
- transform-change invalidation

Validation:
- change master gravity over time
- rotate child simulation frame
- verify propagated local gravity updates

## Milestone 4 — field source schema + CoM coupling
Implement:
- `FieldSourceStore`
- basic cutoff policy
- CoM-only field coupling for one source type

Validation:
- static/dynamic field-source tests
- CoM force accumulation

## Milestone 5 — interaction samples
Implement:
- `InteractionSampleStore`
- multi-point body coupling
- torque accumulation from sampled field variation

Validation:
- compare CoM-only vs multi-sample body in a nonuniform field

## Milestone 6 — voxel/scaled-voxel payload prototype
Implement:
- `VoxelChunkStore`
- a minimal property payload example
- optional generation of interaction samples from voxel chunks

Validation:
- a maze wall or heterogeneous object described via voxel/scaled-voxel payload

---

## 10. Deferred items
Do **not** implement yet:
- full volumetric field integration for every body
- non-rigid global simulation frames
- general-purpose voxel collision as the default narrow phase
- high-fidelity EM beyond gameplay-analog fields
- basis-compressed field representations as the first static-field implementation
- pairwise cached dynamic field contribution matrices

---

## 11. Risky assumptions
- assuming body-local non-inertial frames should directly participate in ambient field inheritance like scene frames
- assuming voxelized collision is needed to get different spin response for primitive shapes
- assuming distributed field coupling must begin with full volumetric integration instead of interaction samples
- assuming one geometry representation should serve both simple fast rigid bodies and detailed heterogeneous objects equally well

---

## 12. Immediate next implementation pass
The best next pass after this spec is:

1. implement `RigidBodyStore`
2. implement `CoordinateSystemStore`
3. implement analytic inertia for spheres/boxes
4. implement `ContactPoint` and `ContactManifold`
5. implement a first sequential impulse solver interface
6. implement master-gravity revision propagation
7. implement one `FieldSourceStore` + CoM-only coupling

That sequence keeps the architecture coherent while preserving the fallback-first workflow.
