This is the current high-level resource document for pyAIGameEngine. It is intended to be the single best starting point for architecture, workflow, status, and project operating rules.

- Workspace version: **0.0.7**

## Project direction
pyAIGameEngine is a Python hybrid game engine / physics simulation engine with modular AI components. The project is optimizing early for:
- scalable architecture
- efficient plank-tic update flow
- vectorization and later kernel migration
- preserved compute headroom for AI behavior
- proof-of-concept isolation before promotion into shared engine layers

## Vocabulary currently in force
- **plank-tic**: smallest game/sim time step
- **plank-pix**: smallest distance unit
- **voxellino**: smallest scalar-valued volume element
- **voxel**: smallest cubic boundary/surface element
- **master stationary reference frame**: common coordinate system containing the extent of all others

## Architecture decisions currently active
1. Runtime hot-path data should favor **SoA** layouts.
2. POC1 remains **fallback-first** and should not be blocked by PyBullet.
3. Python runtime baseline is one **single Python 3.12 `.venv`**.
4. AI inference should not be tied to every agent every plank-tic.
5. Repeated object families should use **fixed runtime schemas** and **feature-pack plans** rather than dynamic field fetching.
6. AI scheduling should operate on **model families**, cadence, freshness, and batchability.
7. Shared components should only be promoted out of POCs after validation.

## POC1 validated capabilities
- explicit plank-tic stage loop
- SoA runtime state buffers
- event buffer and event queries
- agent-subset observation building
- simple integrator backend with **NumPy / Numba** runtime modes
- benchmark, rollout, viewer
- per-stage timing/profiling
- render visibility selection / snapshot culling precursor

## AI architecture guidance now considered part of project truth
### Runtime schema / feature-pack direction
- define fixed per-family runtime schemas
- define feature block schemas per model family
- use precomputed feature-pack plans
- keep CPU state authoritative for now
- use block-level freshness/version tracking, not per-scalar tracking

### AI scheduling direction
- schedule by model family, not by individual agent call site
- define cadence, max input staleness, max output staleness, stage deadline, and fallback policy per family
- use CPU pinned feature pack buffers for staged transfer
- use GPU feature store / GPU output store when inference becomes GPU-resident
- overlap CPU work with asynchronous inference where possible

## Bottlenecks to watch
- event-processing Python loops
- feature packing cost
- fragmented copies
- output scattering
- synchronization points between CPU and GPU
- too many tiny model launches
