# pyAIGameEngine — Extended Project Instructions

This project is for the conceptual design, research, implementation design, and program design of a Python-based hybrid between a game engine and a physics simulation engine, with eventual modular AI integration.

The project goal is to build an engine architecture that is efficient enough per update cycle to support increasingly sophisticated game mechanics and AI-driven behavior without requiring a ground-up refactor when complexity grows. The initial proof of concept may be simple, but the architecture must be designed from the start for extensibility, performance, and maintainability.

## Core project concepts and vocabulary

Use the following terms consistently throughout the project unless explicitly revised later:

- **plank-tic**: the smallest unit of time in the simulation/game. All in-simulation time differences are integer numbers of plank-tics. “Frame rate” in this project context is the frequency of plank-tics per clock second.
- **plank-pix**: the smallest possible unit of distance in the game space.
- **voxellino**: the smallest unit of volume in the engine; effectively a `ds x ds x ds` cube where `ds` may be a plank-pix. A voxellino may only hold scalar values for properties.
- **voxel**: the smallest cubic volume used as an atomic surface/boundary description element, consisting of an integer number of plank-pix on a side and containing a plank-pix³ number of voxellinos.
- **coordinate system**: a perfectly packed set of voxels containing integer numbers of voxellinos.
- **master stationary reference frame**: the coordinate system whose voxels contain the extent of all other voxels in the game space.
- **shape/object**: an entity defined relative to a coordinate system that includes a surface/boundary description and a full description of the properties of any voxel that contains part of the shape.

Unless later revised, assume the working boundary model is:
- a voxel’s boundary is represented internally by two angular values and one length value defining an oriented plane within the voxel;
- points “above” the plane are outside the surface and points “below” the plane and inside the cube are inside the boundary;
- if the length value is greater than `sqrt(3)`, the entire voxel is inside the boundary;
- if the length value is `<= 0`, the entire voxel is outside the boundary;
- continuous properties within a voxel may be approximated using linear gradients between values on the six voxel faces, or an equivalent linear-coefficient representation.

## Architectural direction

The engine architecture should prioritize:

- efficient simulation and update of coordinate systems, shapes, and per-voxel or per-voxellino properties;
- efficient comparison/rendering of object descriptions into a common coordinate system where effects, interactions, physics, triggers, and updates are calculated;
- efficient repacking of update results back into object-relative representations;
- performance headroom for AI components that act on game state;
- modular addition of game mechanics, object behaviors, and AI systems;
- a clean path from proof of concept to reusable engine/package.

The engine does not need to begin with full complexity. It may begin with a very simple playable or semi-playable system, such as a cat-and-mouse style proof of concept, but every early implementation decision should be evaluated in terms of whether it supports later complexity without forcing a major redesign.

## AI direction

A major long-term project goal is modular AI integration. The engine should leave enough compute capacity for AI-driven behaviors, such as:

- adversarial or enemy movement and tactics;
- learned decision-making from local state, line of sight, or fog-of-war restricted information;
- predictive or reconstructive game-state inference for obscured regions;
- learned counter-play and adaptive strategy.

Early AI components should favor existing, well-understood model families that are easy to connect, easy to initialize with functional starting weights, and suitable for continued training or reinforcement. Preference should be given to models that can be adapted with minimal friction to game-state inputs and game-action outputs.

## Research-first workflow

Early project work should strongly favor research and existing packages before custom implementation. Research should focus on:

- Python game engines and rendering/game-loop frameworks;
- Python physics simulation tools and geometry/collision libraries;
- array/tensor and high-performance numeric packages;
- profiling and performance-analysis tools;
- vectorization, compilation, JIT, multithreading, multiprocessing, and GPU offloading tools;
- tools for authoring/editing game objects, levels, geometries, and physical properties;
- AI/ML packages and model families suitable for modular game integration.

Whenever research yields meaningful options, record them. Whenever options narrow into a recommendation or decision, record that too. The project should maintain a clear trail from research findings to design choices.

## Project phases to support

The project should generally progress through these phases, while allowing iteration and overlap:

1. Conceptual design and vocabulary stabilization
2. Research into existing tools, libraries, and approaches
3. Engine architecture definition
4. Initial proof-of-concept implementation with minimal mechanics and a simple AI component
5. Controlled sharing with a small group of outside developers
6. Testing and iterative improvement with additional example mechanics and AI elements
7. Window dressing, menus, standard game features, and usability improvements
8. Refactoring and reorganization into a reusable package/framework for building physics-based, AI-augmented games
