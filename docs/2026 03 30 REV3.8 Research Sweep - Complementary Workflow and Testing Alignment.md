## Complementary findings by topic

### 1. Property-based and stateful testing
The prior research emphasized architecture and performance. The complementary testing research strengthens how we should validate that architecture over a long project life.

Recommended insight:
- property-based and stateful testing are a strong fit for contract-heavy engine subsystems whose behavior can be described by invariants rather than only by example cases

High-value current targets:
- scheduler cadence/freshness/fallback/batchability invariants
- packet-lane generation/baseline/supersede invariants
- adapter reset/step/mode/seed metadata invariants
- archival test schema compatibility and append-only behavior invariants

Practical project implication:
- keep explicit acceptance scripts
- add Hypothesis-style invariant tests as a complementary layer, not a replacement for current acceptance/benchmark scripts

### 2. Long-lifecycle benchmark history tracking
The prior research emphasized integrated benchmark interpretation. The complementary benchmark-tooling research suggests a practical refinement:
- ASV is designed to benchmark a single project over its lifetime, keep result data on disk, record machine information, and compare revisions over time

Practical project implication:
- do not replace the current custom harness immediately
- do shape the archival test data and apples-to-apples comparison keys so that they can map cleanly to an ASV-style longitudinal benchmark model later
- preserve benchmark compatibility keys, machine/environment metadata, and explicit stable benchmark subsets now

Recommended near-term additions to archival test data:
- machine identity / environment fingerprint
- benchmark compatibility group key
- benchmark subset classification (smoke / representative / acceptance / regression)
- stable scenario/seed identifiers for apples-to-apples comparisons

### 3. Future experiment tracking systems
The prior research already supported versioning and artifact discipline. The complementary search shows that both DVC and MLflow become more valuable when the project grows into many model runs, artifacts, and comparisons.

Practical project implication:
- DVC is the stronger conceptual match for "experiments as code" and versioned reproducibility tied to Git
- MLflow is a stronger candidate for richer run UIs and later model/metric comparison
- neither is recommended as a required dependency now

Recommended stance:
- keep the current lightweight artifact/versioning plan for now
- design the archival schema so it can later export to or coexist with DVC/MLflow-style run metadata
- defer adopting either system until AI/RL experiment count, artifact volume, or collaboration pressure justifies it

### 4. Adapter refinement from official environment guidance
The prior research only lightly touched environment wrappers. Complementary Gymnasium guidance supports:
- explicit seeding at reset
- returning observation plus info on reset
- keeping environment metadata and modes explicit

Practical project implication:
- keep the adapter thin and non-authoritative
- record seed and mode as first-class run metadata
- preserve a clean difference between train / eval / benchmark modes in both runtime behavior and archival test data

## Recommended now
- Preserve SoA-first, fallback-first, family-scheduled architecture.
- Add property-based/stateful invariant tests as a complementary validation layer for scheduler, packet, adapter, and archival schema behavior.
- Extend the test/reporting design so the archival benchmark history includes compatibility keys, machine/environment metadata, and stable benchmark subsets.
- Keep the future adapter boundary aligned with Gymnasium reset/seed/info expectations without letting the wrapper become authoritative.
- Update prompts/templates so research sweeps explicitly include complementary search topics beyond the initial architecture survey when useful.

## Plausible later
- Evaluate ASV as a dedicated benchmark-history frontend or export target once the benchmark suite stabilizes.
- Evaluate DVC for experiment-as-code reproducibility once scenario bundles, models, and test artifacts multiply.
- Evaluate MLflow once richer run-comparison UI and model/result browsing become a bottleneck.
- Revisit AoSoA / GPU / chunked scheduling only after current test/reporting and adapter-facing gates are better established.

## Deferred / not yet
- Replacing the current benchmark harness with ASV right now
- Adopting DVC or MLflow as required project infrastructure right now
- Expanding to full RL training workflows before the testing/reporting system and adapter/report metadata are stronger
- Treating Gymnasium convenience concerns as reasons to reshape the authoritative engine path

## Documentation and workflow implications
### Workflow refinements
- Research sweeps should name at least one complementary topic beyond the main architecture topic when the project is broadening.
- Acceptance scripts should remain the primary gate-clearing evidence.
- Property-based/stateful tests should be introduced as reusable invariant checks after or alongside acceptance scripts, not instead of them.
- Apples-to-apples benchmark history should explicitly distinguish:
  - smoke
  - representative
  - regression
  - acceptance-linked benchmark subsets

### Prompt refinements
- testing-system implementation prompts should explicitly preserve append-only archival truth and regenerable report outputs
- research prompts should ask for recommended now / plausible later / deferred categorization
- implementation prompts for test/reporting should ask for compatibility keys and future export readiness, not only raw metric storage
