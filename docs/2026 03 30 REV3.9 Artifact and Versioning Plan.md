# 2026 03 30 REV3.9 Artifact and Versioning Plan

## Purpose
This document defines the lightweight artifact/versioning discipline needed before pyAIGameEngine widens into broader experiment, simulation, and AI/RL work.

## Artifact classes currently in scope
- candidate snapshots
- benchmark configurations
- scenario definitions / bundles
- model artifacts later
- hyperparameter / experiment configs later
- adapter-facing run metadata now
- archival test history now

## Adapter-facing and test-run fields required now
Current adapter-facing and archival test runs should record at minimum:
- `scenario_version`
- `adapter_version`
- `seed`
- `baseline_policy_id`
- `mode`
- `compatibility_key`
- machine/runtime metadata where practical

Recommended near-term additions:
- `scenario_bundle_id`
- `observation_contract_version`
- `action_contract_version`
- `benchmark_subset_id`
- `harness_schema_version`

## File naming guidance
Recommended lightweight conventions:
- scenario bundle: `scenario_bundle_<name>_v<major.minor>.json`
- adapter config: `adapter_config_<name>_v<major.minor>.json`
- baseline policy ID: `baseline_<name>_v<major.minor>`
- archival test history: `test_run_archive_<suite>.jsonl`
- current report: `current_test_report_<suite>.md`

## Current recommendation
Do not build a heavy artifact-management system yet.
Do start treating:
- adapter-facing mode tags
- seeds
- scenario versions
- baseline policy IDs
- compatibility keys
- benchmark subset identifiers
as first-class recorded artifacts.

## Plausible later
- export or map archival benchmark history to ASV-style history
- use DVC/MLflow once experiment scale justifies them
