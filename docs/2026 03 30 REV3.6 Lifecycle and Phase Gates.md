# 2026 03 30 REV3.6 Lifecycle and Phase Gates

## Purpose
This document adds a lightweight lifecycle-management layer to pyAIGameEngine so progress is measured by explicit phase gates, not only by implemented features or benchmark results.

## Lifecycle management principles in force
- Keep the project **restartable** at every meaningful stopping point.
- Require clear **entry / exit criteria** for each near-term phase.
- Treat **benchmark status**, **verification status**, **simulation credibility status**, and **artifact/configuration status** as separate truth sources.
- Prefer small, phase-gated expansions over broad multi-subsystem jumps.
- Use candidate snapshots as baseline checkpoints before widening implementation scope.

## Current lifecycle position
The project is currently in:
**Phase 3 -> Phase 4 transition**

Meaning:
- architecture foundation work is established
- fallback-first POC behavior is validated
- scheduler and packet acceptance have been defined as a completed prerequisite for widening AI/RL-facing work
- the next gate is the first controlled environment-adapter and baseline-policy boundary

## Phase model

### Phase 0 — Governance bootstrap
**Goal:** make the project restartable, traceable, and manageable.

**Status:** substantially complete for current scale.

### Phase 1 — Architecture foundation and fallback POC
**Goal:** establish the authoritative runtime direction and prove the fallback path.

**Status:** achieved for the current POC branch.

### Phase 2 — Verification and simulation-credibility foundation
**Goal:** ensure the engine is not merely evolving, but being checked against explicit claims.

**Status:** active and in force through the governance documents.

### Phase 3 — AI scheduling and interface stabilization
**Goal:** stabilize the engine-facing AI interfaces before broad model complexity.

#### Gate D — Scheduler and Packet Acceptance Gate
**Status:** cleared based on explicit acceptance evidence.

Cleared scope:
- cadence
- freshness
- fallback policy
- batchability
- packet-lane semantics for replace / delta / no_change / delete

### Phase 4 — Early AI/RL experimentation
**Goal:** support controlled AI and RL experiments without destabilizing the core engine.

#### Gate A — Environment Adapter and Baseline Policy Readiness
**Status:** next gate to clear.

**Acceptance intent:**
- minimal non-authoritative environment adapter exists
- explicit `train` / `eval` / `benchmark` mode tagging exists
- first baseline policies exist (hold + simple reflex)
- adapter reports required artifact fields
- adapter respects contract-facing observations and accepted action boundary

### Phase 5 — Broader benchmark matrix and candidate milestone hardening
**Goal:** validate scale and maintainability before collaboration or package shaping.

### Phase 6 — Candidate milestone / relay-ready collaboration point
**Goal:** prepare a clean candidate branch that can be relayed, reviewed, or shared safely.

## Current recommendation
The next implementation pass should explicitly try to clear **Phase 4 Gate A**, not widen the project in multiple directions at once.
