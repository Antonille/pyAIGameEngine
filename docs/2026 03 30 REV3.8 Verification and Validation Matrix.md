# 2026 03 30 REV3.8 Verification and Validation Matrix

## Purpose
This matrix defines what pyAIGameEngine currently claims, how those claims are checked, and what evidence is expected before the project treats a subsystem as accepted for the current phase.

## Current V&V posture
The matrix distinguishes:
- engine/runtime verification
- packet/action boundary verification
- benchmark tooling verification
- simulation behavior verification
- AI scheduling verification
- environment-adapter readiness verification
- long-lifecycle test/reporting-system verification

## Matrix

| Area | Current claim | Verification / validation method | Evidence currently expected |
|---|---|---|---|
| Engine/runtime behavior | fallback-first CPU path remains authoritative and functional | repeatable local validation commands | rigid-body/field prototype pass; smoke benchmark pass |
| Packet/action boundary | multi-family packet boundary is explicit and benchmark-visible | packet logs + acceptance checks | packet metrics; accepted replace/delta/no_change/delete semantics |
| Benchmark tooling | smoke vs representative subsets are distinguishable | benchmark compare workflow | medium/long representative compare results; smoke instrumentation run |
| Simulation behavior | current field / rigid-body subset behaves as expected for prototype scope | prototype validation script | rigid-body / gravity / field validation output |
| AI scheduling behavior | cadence/freshness/fallback/batchability are explicit and accepted | Gate D acceptance script | passing schedule and batchability checks |
| Environment adapter readiness | thin adapter can be added without becoming authoritative | Gate A acceptance evidence | adapter reset/step path, mode tags, baseline policies, artifact fields |
| Long-lifecycle testing/reporting system | append-only archival truth and regenerable reports can be introduced without destabilizing current workflow | future implementation pass + schema validation + report regeneration checks | archival run record written; compatibility keys present; current report scaffold regenerated |

## Research-aligned V&V refinement
Future validation should add invariant-oriented checks for:
- scheduler semantics
- packet-lane generation/baseline/supersede rules
- archival append-only behavior
- compatibility-group-based apples-to-apples comparison rules

Property-based/stateful tests are recommended as a complementary layer once the first custom harness slice exists.
