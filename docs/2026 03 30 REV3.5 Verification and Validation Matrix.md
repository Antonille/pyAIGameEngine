# 2026 03 30 REV3.5 Verification and Validation Matrix

## Purpose
This document defines what pyAIGameEngine is currently claiming, how those claims are checked, and what evidence is considered sufficient for the current project phase.

## Interpretation
- **Verification** asks whether the implemented system matches the intended design/spec.
- **Validation** asks whether the behavior is acceptable for the current intended use.
- At the current phase, many entries are still **prototype-valid for intended internal use**, not general-purpose validated software claims.

## Current acceptance posture
For the current phase, a subsystem is acceptable when:
- it behaves consistently under the documented validation commands
- its current claim is explicitly documented
- its limits / intended-use scope are explicit
- it does not force premature commitment beyond current project truth

## Gate D evidence now in force
Explicit prototype-scope acceptance evidence now exists for:
- scheduler cadence
- scheduler freshness metadata
- scheduler fallback policy metadata
- transfer batch splitting / batchability
- packet-lane `replace` behavior
- packet-lane `delta` baseline behavior
- packet-lane `no_change` cache-hit behavior
- packet-lane `delete` supersede behavior
- output-staleness-triggered hold-family refresh behavior

## Current gaps
The following are still missing or only partially covered:
- broader negative tests for ownership-lane edge cases
- simulation validation beyond the present prototype scenarios
- explicit uncertainty / sensitivity recording for simulation parameters
- formalized RL environment adapter verification

## Usage guidance
When adding or revising a subsystem:
1. update the claim in this matrix
2. record the evidence that currently supports it
3. say whether the claim is verified, validated for limited internal use, or still exploratory
4. name the next strengthening step
