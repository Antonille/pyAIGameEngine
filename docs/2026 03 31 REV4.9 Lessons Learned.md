# 2026 03 31 REV4.9 Lessons Learned

## Purpose
Capture durable project and workflow lessons after adding the first startup-compiled configurable-interface groundwork slice.

## Best practices strengthened
- Keep the active repo/documentation surface intentionally small.
- Archive superseded revision series instead of letting them compete with current truth.
- Treat configurable interfaces as boundary-control work before broad runtime refactors.
- Preserve simulation-authoritative structures separately from interface-facing structures.
- Keep archival test data durable and current reports regenerable.

## New lessons from the configurable-interface groundwork pass
### Best practice now adopted
A configurable interface should be introduced as a startup-loaded, startup-validated, startup-compiled plan before any attempt to replace hot-path runtime mappings.

### Best practice now adopted
The configurable interface must define interface points clearly: what they are, where they are, and what they do. Boundary and exclusion zones are part of the contract, not optional commentary.

### Best practice now adopted
The readable interface document and the programmatic validator must be synchronized from one canonical source whenever practical. Manual two-source sync should be treated as a drift risk.

### Best practice now adopted
The first bounded slice should prove the boundary on one repeated-family hot control-plane subset before widening to richer channel roles or representation kinds.

## Open risks still visible
- the current implementation only covers a narrow hot observation/action subset
- reward/done/memory/report/latent/narration channels remain deferred
- binary manifest serialization remains deferred until the semantic contract stabilizes
- broader model-builder/toolchain alignment is not yet implemented
