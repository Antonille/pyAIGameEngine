# 2026 03 31 REV4.8 Test Report Procedure

## Purpose
This document formalizes the project's named **Test Report Procedure**.

It converts the original long-form implementation-and-workflow prompt for the long-lifecycle integrated testing and reporting system into an active reusable procedure for future:
- testing passes
- testing-analysis/report passes
- report-regeneration passes
- validation-and-review passes

## Procedure status
- status: **active named procedure**
- applies to: **POC1_SoaFirst** and later engine-level testing/reporting systems unless superseded
- truth rule: **archival test data remains durable truth; current human-readable reports remain regenerable**

## Controlling intent
The test/reporting system should support a long engineering life cycle with:
- append-only archival truth
- modular but integrated test execution
- regenerable current reports
- apples-to-apples comparison over time
- provisional performance-versus-complexity projection
- future smoke / acceptance / regression / benchmark / simulation-credibility / AI-RL-facing expansion

## Core procedure rules

### 1. Truth rule
- archival test data is append-only and durable
- current human-readable reports are replaceable/regenerable
- reports must remain reproducible from archival test data plus referenced stored artifacts
- supporting local observations do not override archival or repo truth unless explicitly accepted and promoted

### 2. Execution rule
While implementing or refining the testing/reporting system, evaluate whether the current design remains optimal.
If a better design is discovered during implementation, then:
- document the change
- update the plan
- implement the improved design in the same pass when safely bounded

### 3. Scope rule
The system must support future:
- smoke tests
- acceptance tests
- regression tests
- benchmark/performance tests
- simulation-credibility tests
- AI/RL-facing tests

### 4. Prompting rule
Minimize terminal prompting needed to launch the desired test flow.
Durable local calling syntax should remain simple, repeatable, and easy to hand off.

## Required architecture intent

### Integrated harness
The testing system should provide a modular but integrated test harness that can:
1. load config
2. collect metadata
3. run one or more suites
4. append results to archival data
5. regenerate current report products
6. emit concise terminal summaries with artifact paths

### Archival data
Archival test data must preserve enough metadata to recreate current reports, including where applicable:
- test run identifier
- timestamp
- project / snapshot identity
- harness / schema version
- environment / mode metadata
- executed test groups
- parameters used
- validation results
- performance results
- comparison hooks to prior compatible runs
- assumptions used for projection metrics
- references to generated artifacts / images / plots

### Comparison/projection
The procedure should preserve:
- apples-to-apples comparison capability
- raw performance-over-time tracking
- a complexity-normalized projection method or its current best bounded approximation
- notes explaining when the current metric is temporarily unrepresentative because of architecture churn

## Required human-readable report content

Do not remove the following content classes. Formatting and section naming may improve over time, but the output must still contain their substance.

### A. Human-oriented consumer summary
Include:
- summary of current test data
- re-validation of previous success/failure metrics
- summary of new success/failure metrics
- performance tests and results
- visualization of tests
- key rendered simulation snapshots where relevant, labeled and described for significance

### B. Numerical and visual comparison toward future performance
Include:
- raw performance vs time
- apples-to-apples comparison traceable over development
- a metric estimating performance / complexity
- projection guidance toward estimated future engine performance
- notes on large fluctuations, including:
  - acknowledgement if improved
  - improvement plan if worsened
  - explanation when the current metric is known to be temporarily unrepresentative

### C. Appendix
Include:
- current state of development (implemented vs unimplemented)
- table of test parameters used
- brief justification of assumptions made
- compatibility notes where useful
- artifact list where useful

## Workflow intent
The intended workflow remains:

1. provide calling syntax for the test program when appropriate
2. run the test program locally
3. upload resulting archival test data and generated artifacts when analysis/regeneration is needed outside the local run
4. validate and analyze returned test results
5. integrate accepted test data into the next snapshot unless the run is invalid or must be repeated
6. regenerate:
   - the current human-readable report
   - updated comparisons
   - updated tracking and snapshot artifacts as needed

## Analysis-pass rule
When doing a testing-analysis/report pass:
- do not merely check that files exist
- evaluate whether the current report and artifacts satisfy the intended report structure
- distinguish clearly between:
  - durable archival truth
  - regenerable current report products
  - auxiliary/legacy convenience visuals
- identify whether any stale/legacy outputs are mixed into the current report surface and should be archived, removed, or explicitly labeled

## Implementation-pass rule
When doing a testing/reporting implementation pass:
- preserve the durable-vs-regenerable separation
- avoid overwriting archival history
- allow overwrite/regeneration of only the current report products
- leave exact local calling syntax
- leave exact validation commands
- update current docs/tracking when the procedure meaningfully changes

## Documentation update rule
When project docs discuss testing/report generation, they should reflect this named procedure where relevant:
- workflow docs
- resource guide
- lessons learned
- prompt templates
- next-step prompts
- active file/path docs when report products change

## Current caution
This procedure is now active project guidance, but not every current repo doc may yet be perfectly aligned with it.
When a doc conflicts with this procedure and no newer accepted replacement exists, update the doc rather than silently drifting from the procedure.
