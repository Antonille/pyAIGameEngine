# 2026 03 30 REV4.3 AI Interface Control

## Purpose
Define the current control boundary between pyAIGameEngine and AI/RL-facing adapters.

## Decided now
- The authoritative simulation clock is the plank-tic loop.
- Observation exposure should be staged from SoA-authoritative state using runtime-schema-driven feature blocks.
- The authoritative control path is:
  1. action packets
  2. action-bridge application
  3. authoritative write into `state.action`
  4. downstream action application / physics consumption
- Thin adapters may submit packets, but should not become the engine's public AI API.
- Timing visible at the adapter boundary should expose input-stage packet application timing, AI-stage packet build timing, action application timing, and event/observation timing.

## Temporary scaffolding
- The minimal Gym-compatible env remains a project-local scaffolding path.
- Narrow POC-local observation helpers still exist, but should be interpreted as staging helpers rather than the long-term public contract.

## Deferred
- Public RL framework API hardening.
- Many-world / headless parallel rollout as a current milestone constraint.
- Framework-specific model-serving contracts in the hot simulation core.

## Open questions
- Minimum stable feature-block set for the first explicit adapter contract.
- Minimum stable action packet metadata needed for external controllers beyond the current POC.
- Whether adapter-level timing should later include transfer staging bytes as a first-class surfaced metric.
