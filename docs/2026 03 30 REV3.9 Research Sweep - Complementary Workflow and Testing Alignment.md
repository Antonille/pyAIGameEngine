## Complementary findings
### 1. Invariant-oriented testing is a strong fit for this project
Property-based testing is well-suited for invariants that are difficult to cover exhaustively with hand-written examples, and Hypothesis explicitly supports both standard property tests and stateful tests that generate sequences of operations rather than single inputs.

Research-aligned implication for pyAIGameEngine:
- Use explicit acceptance scripts for named gates **now**
- Add **property-based / stateful invariant tests later** for:
  - scheduler cadence and freshness invariants
  - packet-lane ownership and supersede semantics
  - archival schema compatibility invariants
  - adapter reset / step / mode-tag invariants

### 2. Long-lifecycle benchmark history should be treated as a first-class requirement
ASV is designed to benchmark a single project over its lifetime, stores result data as JSON alongside the benchmark suite, and is built around comparable benchmark suites rather than ad hoc individual timing runs.

Research-aligned implication for pyAIGameEngine:
- Keep the current lightweight benchmark scripts now
- Design archival benchmark/test data so it can later export to or interoperate with an ASV-like history model
- Record compatibility keys, machine/environment metadata, and stable benchmark subset identifiers now so apples-to-apples comparison remains possible later

### 3. Adapter-facing experiment tracking should start lightweight
Gymnasium guidance reinforces explicit seeding at `reset()`, observation-plus-info return shape, and explicit environment checking. DVC experiment tracking reinforces that parameters, metrics, artifacts, and reproducibility metadata should be tracked together, but a full experiment-management platform is not yet required for current scope.

Research-aligned implication for pyAIGameEngine:
- For current adapter-facing work, record:
  - `scenario_version`
  - `adapter_version`
  - `seed`
  - `baseline_policy_id`
  - `mode`
  - machine/runtime metadata where practical
- Do **not** adopt DVC/MLflow immediately
- Keep the first testing/reporting implementation append-only and file-based, but structure it so later experiment tooling remains possible

## Recommended now
- Keep the SoA-first / fallback-first architecture path
- Keep NumPy + Numba as the current performance path
- Keep packet/action interfaces explicit and versioned
- Add append-only archival test data with:
  - compatibility keys
  - machine/environment metadata
  - stable benchmark subset identifiers
  - artifact references
- Keep the adapter boundary thin and non-authoritative
- Keep benchmark and report regeneration grounded in fixed seeds and explicit mode tags

## Plausible later
- Hypothesis property/stateful invariant suites for scheduler, packet, adapter, and archival-schema behavior
- ASV as a benchmark-history backend or export target
- DVC or MLflow when experiment/artifact scale justifies them
- More advanced scheduling variants such as chunked or disaggregated batching, once the current baseline is stable and measured

## Deferred / not yet
- Replacing the current benchmark scripts immediately
- Heavy experiment-management infrastructure now
- GPU-first or multi-process architecture changes before the CPU-authoritative path and archival test/reporting loop are stable
- Broader RL/training expansion before the testing/reporting system exists

## Workflow and prompt implications
- Research sweeps should include at least **two complementary topics not already covered** by the attached deep research
- Benchmark-related prompts should ask for:
  - fixed seeds
  - compatibility keys
  - machine/runtime metadata
  - explicit apples-to-apples subset naming
- Testing/reporting prompts should distinguish:
  - archival truth (append-only)
  - current report (regenerable)
- Implementation prompts should explicitly state whether new tooling is:
  - recommended now
  - plausible later
  - deferred

## Current takeaway
The complementary research supports the current architecture path. The strongest improvements are in:
- testing rigor
- benchmark-history discipline
- archival/reporting structure
- adapter-facing metadata and seeding discipline

It does **not** justify an architecture pivot away from the current fallback-first SoA path.
