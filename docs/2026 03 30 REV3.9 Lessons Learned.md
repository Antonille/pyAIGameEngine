# 2026 03 30 REV3.9 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, prompting, and governance lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Revisioned markdown files are enough for early tracking; do not overbuild project-management tooling yet.
- Root-level install, snapshot, and GitHub scripts reduce restart friction.
- Baseline the project on **Python 3.12.x** for now to reduce package friction across PyBullet, ModernGL, Gymnasium, Numba, and Torch.
- A single `.venv` is simpler than split environments at this stage of the project.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Integrated benchmark/runtime truth should outweigh isolated microbenchmark wins when current evidence is available.
- Smoke runs are for instrumentation visibility; representative runs should drive benchmark claims.
- Benchmark outputs should increasingly carry fixed seeds, compatibility identifiers, and machine/runtime metadata so apples-to-apples comparison remains possible over time.

## Research-aligned additions from the complementary sweep
- Hypothesis-style property testing is a strong future fit for scheduler, packet, adapter, and archival-schema invariants.
- Hypothesis stateful testing is especially relevant when correctness depends on **sequences of operations** rather than one isolated input.
- Long-lifecycle benchmark tracking benefits from a stable benchmark suite, stable comparison subsets, and durable result files; future outputs should stay compatible with that style of history even if current scripts remain lightweight.
- Adapter-facing environment paths should make seeding and `reset()` discipline explicit and preserve observation-plus-info style interfaces.
- Experiment tracking platforms are plausible later, but append-only local archival test data is the correct near-term foundation.
- Research sweeps are more useful when they explicitly seek at least **two complementary topics not already covered** by the attached research input.

## Prompt and workflow lessons already in force
- Prompts are more effective when the pass type is explicit.
- The most reliable prompts do two things together:
  - anchor to current validated truth
  - explicitly protect against premature overcommitment
- Exact local validation commands, expected outputs, and the next recommended prompt create a tighter and more productive loop.
- Interface/control docs and lightweight façade modules should come before a larger refactor.
- Simulation-authoritative structures should not automatically become public interface structures.
- “What not to build yet” should be recorded alongside “what to build now.”
- Optimized prompts are part of the project operating system, not just convenience text.
- Relay suggestions are better as soft triggers than forced interruptions.

## Current takeaway
The complementary research did **not** change core runtime truth. It strengthened the case for:
- append-only archival testing/reporting
- fixed-seed benchmark discipline
- compatibility-key based comparison
- thin, explicit adapter-facing metadata
- future invariant/property testing
