# 2026 03 30 REV4.1 GitHub and Snapshot Workflow

## Purpose
Keep the local repo, the GitHub repo, snapshot exports, and project-truth docs aligned.

## Canonical repo identity
- GitHub repo: `Antonille/pyAIGameEngine`
- Remote URL: `https://github.com/Antonille/pyAIGameEngine.git`
- Expected default branch: `main`
- Visibility: `private`

## Transition-state rule
The repo now exists, but a newly created or empty repo does not automatically become code truth.

Use this rule:
- if the intended baseline is committed and pushed, GitHub is the primary shared truth
- if the local tree is newer and unpushed, the local tree is immediate truth and must be pushed or explicitly snapshotted/reconciled
- if a chat uses an uploaded candidate snapshot that is newer than the repo, that snapshot becomes immediate truth for the pass until reconciled back into GitHub

## Standard stopping-point routine
1. run validation commands
2. update current docs and manifest/history
3. review `git status`
4. commit with a meaningful message
5. push the intended branch
6. optionally export sparse or full snapshot
7. if snapshots or uploaded run outputs were used in the pass, make sure the repo catches back up

## Recommended local setup / conversion commands
If the local repo is not yet connected to the new remote:
```powershell
git remote remove origin
git remote add origin https://github.com/Antonille/pyAIGameEngine.git
git branch -M main
git status
```

If the repo is newly initialized locally:
```powershell
git init
git remote add origin https://github.com/Antonille/pyAIGameEngine.git
git branch -M main
git add .
git commit -m "Initialize pyAIGameEngine baseline"
git push -u origin main
```

If git is already initialized locally and you are pushing the current baseline:
```powershell
git status
git add .
git commit -m "Align pyAIGameEngine baseline with GitHub"
git push -u origin main
```

## Snapshot role from this point forward
Snapshots remain important, but their role changes:
- archive / rollback
- relay / chat transfer
- large sparse refresh handoff
- preservation of generated artifacts and result bundles
- recovery when local work is ahead of the repo

Snapshots are no longer the default day-to-day authoritative source once the repo baseline is active.

## Snapshot naming
- sparse: `pyAIGameEngine_sparse_snapshot_<date>_<rev>_<topic>.zip`
- full: `pyAIGameEngine_full_snapshot_<date>_<timestamp>.zip` or equivalent clean milestone naming

## Snapshot metadata fields recommended now
When practical, include:
- repo full name
- branch
- commit sha
- whether local work was ahead of the repo at snapshot time

## Clean snapshot exclusions
Do not include:
- `.venv`
- `__pycache__`
- `.egg-info`
- `.pytest_cache`, `.mypy_cache`, `.ruff_cache`
- numba cache artifacts
- `build`, `dist`, transient output folders
- large generated outputs unless they are intentionally part of the handoff
