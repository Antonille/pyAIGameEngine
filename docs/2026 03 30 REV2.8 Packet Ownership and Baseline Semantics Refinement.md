# 2026 03 30 REV2.8 Packet Ownership and Baseline Semantics Refinement

## Purpose
Record the implementation pass that refined packet ownership and baseline semantics after the explicit `no_change` workload had a recorded benchmark subset.

## What changed
- `AIActionPacket` now carries:
  - `ownership_scope`
  - `ownership_key`
- packet protocol version advanced to `action_packet_v3`
- `ActionBridge` now distinguishes:
  - packet cache reuse identity
  - packet ownership-lane identity
- ownership lanes are tracked per:
  - source family
  - runtime family
  - action block
  - batch slot
- a concrete ownership key is derived from the lane scope plus the current entity-batch digest
- replace packets can explicitly supersede the prior lane state
- delete and delta packets now carry clearer baseline/supersede references without forcing every refresh through a separate invalidate packet
- packet summaries and logs now surface ownership scope/key information for easier debugging

## Why this was done
The earlier packet boundary had baseline and supersede references, but cache-key matching and ownership reasoning were still entangled. This refinement separates those concerns while keeping the boundary contract-facing.

## Consequences
- Packet logs are more informative.
- Ownership transitions are easier to inspect.
- Packet counts can decrease compared with the earlier implementation because delete/delta refreshes no longer need to emit as many explicit invalidate packets.
- The benchmark truth should be refreshed before any new winner claim is recorded for this refined workload.
