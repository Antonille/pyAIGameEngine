# 2026 03 29 REV1.7 AI Developer Guide

## What this guide is for
This guide explains how to think about the engine from the **AI developer** side.

You are responsible for:
- supplying AI model-family implementations
- consuming designated feature blocks
- producing actions/decisions/latent memory
- defining reward shaping or experiment-specific learning logic

You are **not** expected to know:
- collision solver internals
- field atlas internals
- renderer internals
- low-level physics backend implementation details

## Mental model
The engine provides:
- scheduled opportunities to run model families
- designated feature blocks
- canonical objective/event signals
- action application semantics

You provide:
- inference logic
- memory/state logic
- reward shaping logic
- training/evaluation logic

## Reward handling
Current recommended split is **hybrid**:
- engine emits canonical objective/event signals
- AI backend composes or shapes rewards from those signals

This keeps experimentation flexible without losing standardization.

## Near-term best practices
1. Build against feature-block contracts, not raw engine internals.
2. Treat actor-family indexing as explicit and stable.
3. Keep action packet semantics simple and versionable.
4. Keep reward shaping logic separate from engine solver code.
5. Expect scheduler decisions to depend on cadence, freshness, and fallback policy.

## What to expect from the engine
You should expect:
- feature-block definitions
- action/output contracts
- reward-signal contracts
- family/cadence/freshness metadata

## What not to assume
Do not assume:
- every actor runs inference every plank-tic
- raw solver state is your public API
- reward logic is fully embedded in the engine
- GPU staging details are final yet

## Near-term deliverable expectation
The near-term goal is an AI-facing contract and lightweight façade, not yet a finalized independent AI package.
