# 2026 03 30 REV3.7 Apples-to-Apples Comparison and Projection Method

## Purpose
This document defines the first apples-to-apples comparison rules and a provisional complexity-normalized performance projection method for pyAIGameEngine.

## Apples-to-apples rule
A run is comparable only when the comparison key matches on all required fields.

## Required comparison fields
- benchmark family id
- backend mode
- scenario bundle id or scenario family
- adapter version if adapter-facing
- baseline policy id if used
- mode (`train`, `eval`, `benchmark`)
- bodies
- steps
- feature family set
- packet mode set
- seed policy compatibility

If these differ materially, the runs should not be treated as direct trend points in the same apples-to-apples series.

## Trend rule
Maintain at least one stable benchmark family specifically for long-term trend tracking. This benchmark family should be intentionally conservative about parameter changes.

## Projection principle
The projection method is provisional and assumption-heavy. It must be presented as an estimate, not as a precise forecast.

## Proposed complexity-normalized metric
Define:
- `current_measured_ms_per_step`
- `current_complexity_fraction`

Where `current_complexity_fraction` is the weighted estimated fraction of the intended final baseline computational load currently represented.

### Proposed component weights
Early example weights:
- physics / simulation core
- scheduler / packet system
- adapter / baseline policy layer
- reporting / instrumentation overhead
- future baseline RL agent inference allowance

Each component gets:
- `target_weight`
- `current_realization_fraction` in `[0,1]`

Then:
`current_complexity_fraction = sum(target_weight_i * realization_fraction_i)`

### Provisional normalized metric
`normalized_ms_per_full_complexity = current_measured_ms_per_step / max(current_complexity_fraction, epsilon)`

## Usage rule
- freeze the initial weighting assumptions once adopted
- revise only by explicit documented decision
- trend the normalized metric over time
- note when a stage is clearly non-representative and should not be overinterpreted

## Reporting rule
Always show:
- raw measured performance
- apples-to-apples compatible trend
- complexity-normalized estimate
- assumptions used
