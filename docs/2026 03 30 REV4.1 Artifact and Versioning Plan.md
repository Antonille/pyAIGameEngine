# 2026 03 30 REV4.1 Artifact and Versioning Plan

## Purpose
This document defines the lightweight artifact/versioning discipline needed before pyAIGameEngine widens into broader experiment, simulation, and AI/RL work.

## GitHub-linked versioning rule
When the repo baseline is active, GitHub commit history becomes the primary durable version trail for:
- code state
- docs state
- workflow changes
- reviewable milestones

Snapshots remain secondary artifacts for archive, relay, rollback, and handoff.

## Snapshot package metadata required now
Every full, sparse, and candidate snapshot should include:
- `OPEN_THIS_FIRST.md`
- `SNAPSHOT_INDEX.json`
- `docs/CURRENT_ACTIVE_FILE_PATHS.md`

These files exist to speed truth establishment and reduce reliance on conversational memory.

## Snapshot metadata fields recommended now
`SNAPSHOT_INDEX.json` should record at minimum:
- `snapshot_type`
- `snapshot_timestamp`
- `project_root_in_zip`
- `workspace_version`
- `poc_package_version`
- `current_pointer_docs`
- `active_files`
- `recently_modified_files`
- `manifest_file`
- `modification_history_file`
- `next_prompt_file`
- `git_repo`
- `git_branch`
- `git_commit`
- `notes`

## Active-path tracking required now
`docs/CURRENT_ACTIVE_FILE_PATHS.md` should record the current working set for:
- active implementation files
- active docs
- validation scripts
- snapshot/relay scripts
- repo identity
- recently added files
- paths to avoid because they are historical or superseded

## Existing artifact classes currently in scope
- GitHub commits / branches
- candidate snapshots
- benchmark configurations
- scenario definitions / bundles
- model artifacts later
- hyperparameter / experiment configs later
- archival test history now
