# 2026 03 30 REV3.5 Lifecycle and Phase Gates

## Purpose
This document adds a lightweight lifecycle-management layer to pyAIGameEngine so progress is measured by explicit phase gates, not only by implemented features or benchmark results.

## Lifecycle management principles in force
- Keep the project **restartable** at every meaningful stopping point.
- Require clear **entry / exit criteria** for each near-term phase.
- Treat **benchmark status**, **verification status**, **simulation credibility status**, and **artifact/configuration status** as separate truth sources.
- Prefer small, phase-gated expansions over broad multi-subsystem jumps.
- Use candidate snapshots as baseline checkpoints before widening implementation scope.

## Phase model

### Phase 0 — Governance bootstrap
**Goal:** make the project restartable, traceable, and manageable.

**Exit gates:**
- current-truth docs exist
- snapshot workflow exists
- decision/log/lessons workflow exists
- repeatable pass types are in use

**Status:** substantially complete for current scale.

### Phase 1 — Architecture foundation and fallback POC
**Goal:** establish the authoritative runtime direction and prove the fallback path.

**Exit gates:**
- fallback-first CPU path validated
- multi-family runtime schema direction established
- packet/action boundary exists and is exercised
- benchmark tooling can compare realistic integrated workloads

**Status:** achieved for the current POC branch.

### Phase 2 — Verification and simulation-credibility foundation
**Goal:** ensure the engine is not merely evolving, but being checked against explicit claims.

**Exit gates:**
- verification and validation matrix exists
- simulation credibility plan exists
- current benchmark subsets are classified as smoke vs representative
- intended-use scope is documented

**Status:** achieved for current prototype scope.

### Phase 3 — AI scheduling and interface stabilization
**Goal:** stabilize the engine-facing AI interfaces before broad model complexity.

**Exit gates:**
- feature-pack and action-packet contracts are stable enough for structured experimentation
- AI scheduling policies have explicit acceptance checks
- packet ownership/baseline semantics are documented and benchmarked
- at least one AI integration strategy doc exists

**Status:** active; **Gate D cleared** for current prototype scope.

### Phase 4 — Early AI/RL experimentation
**Goal:** support controlled AI and RL experiments without destabilizing the core engine.

**Exit gates:**
- Gymnasium-compatible adapter plan exists
- scripted / heuristic baseline policies exist
- train vs eval separation is documented
- experiment artifacts (seeds, configs, weights, metrics) are versioned

**Status:** not yet active; next gate should prepare entry.

### Phase 5 — Broader benchmark matrix and candidate milestone hardening
**Goal:** validate scale and maintainability before collaboration or package shaping.

**Exit gates:**
- representative benchmark matrix is defined and maintained
- candidate snapshots are easy to rebuild and compare
- lifecycle / V&V / credibility docs remain aligned with code reality

### Phase 6 — Candidate milestone / relay-ready collaboration point
**Goal:** prepare a clean candidate branch that can be relayed, reviewed, or shared safely.

**Exit gates:**
- clean full snapshot candidate exists
- docs, manifest, modification history, and prompts are aligned
- known risks, open questions, and next-step gates are explicit

## How to use this document
At the start of a major pass, ask:
1. which phase am I in?
2. what phase gate am I trying to clear?
3. what evidence counts toward that gate?
4. what should remain deferred until that gate is cleared?
