# 2026 03 31 REV4.6 GitHub and Snapshot Workflow

## Purpose
Clarify how local changes, snapshots, scripts, and line-ending policy should be promoted into stable GitHub truth.

## Current rule
- GitHub main is the authoritative long-term repo-truth surface.
- A full or sparse package may temporarily be ahead of GitHub and should then be treated as bounded `snapshot-truth` pending review and commit.
- Supporting local evidence never overrides repo-truth by itself.

## Canonical local variables
- `path_pyAIGameEngine`
- `path_pyEngine_scripts`
- `path_revs_pyAIGameEngine`

Historical aliases such as `path_zip_revs` and `path_rev_zip` are retired and should not be used in scripts, prompts, or docs.

## Standard promotion workflow
1. Review the target ZIP or local changeset.
2. Replace or update the local repo in a controlled way.
3. Run validation commands.
4. Review `git status` and `git diff --stat`.
5. Review line-ending warnings. If warnings appear unexpectedly, stop and check `.gitattributes` plus the touched files.
6. Commit with a bounded message.
7. Push to GitHub main.
8. Recheck critical public-main entry points.

## Full-snapshot replacement workflow
For cleanup/pruning passes that should delete abandoned files:
1. Use an explicit full-snapshot replacement script or a reviewed mirror command.
2. Preserve `.git` and normally preserve `.venv`.
3. Use dry-run preview before destructive mirror/apply.
4. Confirm that deleted files are genuinely intended to leave repo-truth.

## Line-ending workflow
1. Commit a root `.gitattributes` file.
2. Use repo-level line-ending rules rather than relying on per-user Git defaults.
3. After line-ending policy changes, run:
   - `git add --renormalize .`
   - `git status`
   - `git diff --stat`
4. Commit normalization separately from unrelated feature changes when practical.

## Copy/paste rule
Do not treat copied chat output as the preferred transport for durable repo files.
Prefer one of:
- downloadable file artifact
- ZIP package
- scripted file generation
- direct repo edit with review

If copy/paste is unavoidable, the pasted text must be reviewed in an editor before commit.
