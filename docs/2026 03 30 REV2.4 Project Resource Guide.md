## Research-sweep conclusion
**Yes: the project is still on the right track.**
The strongest research-aligned decisions currently are:
- keep the **CPU-authoritative SoA runtime** as the current truth
- keep **fixed runtime schemas + feature packs + action packets** as the interface boundary
- keep **AI scheduling at the family/cadence/freshness/batchability** level rather than per-agent every plank-tic
- treat **integrated benchmark truth** as authoritative and keep re-checking it as workload mix changes

## Recommended now
- Continue with SoA-first runtime design for hot-path fields.
- Keep sparse or optional payload handles outside the hottest contiguous field blocks.
- Keep packet caching at the **packet boundary**, not the raw model-output boundary.
- Re-check benchmark winner claims whenever family mix, packet width, or cache behavior changes.
- Use the next block to run the benchmark matrix for the packet-cache workload and document the current winner claims with context.

## Plausible later
- Evaluate **AoSoA** layouts if hot-field count grows, vectorization becomes more important, or cache behavior degrades.
- Evaluate **ONNX Runtime** as a model-family serving/runtime layer once model contracts stabilize and cross-language deployment becomes valuable.
- Evaluate **Pythran** or **Cython** only for stable numeric kernels that stop benefiting from Numba or where ahead-of-time/native builds are worth the maintenance overhead.
- Evaluate **CuPy** only after the CPU-authoritative schema, transfer, and packet contracts are stable enough that GPU migration targets are clear.

## Deferred / not yet
- Full PyBullet dependency
- Major repo/package split
- GPU-first runtime ownership
- Early conversion of the runtime from SoA to a more complex AoSoA layout before metrics justify it
- Premature commitment to a single long-term AI inference backend

## Workflow heuristics
- Use a clearly named **pass type** whenever practical: research, research sweep, architecture/spec, implementation, documentation/interface, relay/handoff, or retrospective.
- Start from **current project truth** and explicitly preserve known-good constraints such as fallback-first execution and PyBullet deferral.
- Ask for or deliver **exact validation commands**, expected outcomes, and the next recommended prompt.
- Treat **integrated benchmark/runtime truth** as more authoritative than isolated microbenchmark wins.
- Prefer **interface/control docs and lightweight façades** before major package or boundary refactors.
- Keep **simulation-authoritative** structures separate from **interface-facing** representations.
- Capture both **recommended now** and **deferred / not yet** items so future passes do not overcommit.
- Once cadence-aware scheduling exists, benchmark summaries should include **cumulative run counts**, not only last-cycle observations.
- When a new family, packet contract, or cache policy changes the integrated workload mix, re-check benchmark winner claims before carrying them forward.
- Packet cache hits should be logged at the packet boundary so reuse semantics remain visible without exposing solver internals.
