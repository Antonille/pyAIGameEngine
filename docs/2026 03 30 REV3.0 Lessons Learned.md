# 2026 03 30 REV3.0 Lessons Learned

## Purpose
This document consolidates durable architecture, tooling, benchmarking, workflow, and prompting lessons that should remain easy to recover across chats.

## Architecture and tooling lessons already in force
- Separate proof-of-concept folders reduce accidental cross-talk and make architectural comparison easier.
- The canonical engine-state decision matters early; SoA arrays are the right default experiment for throughput-oriented physics and AI.
- Revisioned markdown files are enough for early tracking; do not overbuild project-management tooling yet.
- Root-level install, snapshot, and GitHub scripts reduce restart friction.
- Baseline the project on **Python 3.12.x** for now to reduce package friction across PyBullet, ModernGL, Gymnasium, Numba, and Torch.
- A single `.venv` is simpler than split environments at this stage of the project.
- Install scripts must create the venv with the intended interpreter, bootstrap `pip` if needed, and verify imports at the end.
- Viewer and benchmark scripts should be runnable independently.
- PyBullet may fall back to source build even on Python 3.12; success depends on a correctly initialized Visual Studio C++ toolchain and Windows SDK environment.
- The install script must never claim success when a package install fails; it must fail fast or explicitly skip optional native packages with a clear warning.
- The engine architecture should preserve a fallback backend path so core SoA / plank-tic work can continue even when optional native dependencies are unavailable.

## Runtime and benchmark lessons already in force
- A standalone kernel speedup does not guarantee an integrated runtime speedup.
- Fair NumPy vs Numba comparison requires warmup awareness and structured benchmark output.
- Separating render-subset selection from full simulation state helps identify render-facing costs and keeps stage boundaries cleaner.
- AI scheduling should start as a minimal spec-driven planning layer before introducing real model execution complexity.
- Full snapshots can drift behind sparse implementation passes; current-state reconciliation should explicitly merge sparse artifacts before publishing a new full snapshot.
- Current-truth pointer docs (`CURRENT_*`) materially reduce restart friction.

## Packet-boundary lessons already in force
### Explicit `no_change` is meaningfully different from packet reuse
A cache hit does not always mean “reapply the cached packet.” Some contracts should deliberately emit a **`no_change` packet-state** so that the packet layer records semantic stability explicitly.

### Cache-hit behavior is contract-dependent
Different output blocks can legitimately use different cache-hit behavior:
- `reuse_packet` for contracts that must keep reapplying a concrete result
- `no_change` for contracts where the engine should preserve the last applied state without resending the payload

### Baseline and supersede references improve traceability
Generation IDs alone are not enough once packet invalidation or richer delta semantics appear. Carrying both a **baseline generation** and a **superseded generation** makes packet history easier to reason about without exposing raw solver internals.

## Ownership-refinement lessons already in force
### Cache-key identity and ownership-lane identity are different concerns
A cache key answers “can this payload be reused?” An ownership lane answers “which packet currently owns this contract-facing action lane?” They often overlap, but they should not be conflated.

### A practical ownership lane can be narrower than a family and broader than a concrete payload
For the current prototype, a lane defined by **family + runtime family + action block + batch slot** is a workable ownership scope. It is stable enough to reason about supersession without exposing lower-level solver structures.

### Replace is not the only packet mode that benefits from baseline metadata
Delete and delta packets also benefit from explicit baseline/supersede references. They should make their relationship to the previous lane state visible even when they do not emit a separate invalidate packet.

### Not every ownership refresh should emit a visible invalidate packet
If a delete or delta packet already semantically supersedes the prior lane state, forcing a separate invalidate packet can overcount packet churn without improving clarity.

## Ownership-targeting and benchmark lessons added in REV3.0
### Ownership-aware semantics can simplify packet metrics materially
Once delete/delta/no-change semantics explicitly identify the owned lane state they relate to, explicit invalidate traffic can legitimately drop to zero for the validated workload. This makes packet counts easier to interpret.

### Packet-layer metrics can stay backend-invariant while integrated winners still change
For the current ownership-aware workload, packet-layer counts match across backends. The backend winner is therefore still being driven by stage timings, especially physics, rather than packet/cache divergence.

### Ownership-aware packet metrics produce a cleaner benchmark subset than the earlier explicit-invalidate-heavy version
The ownership-targeted refinement reduced action-packet churn while preserving semantic traceability. That makes the benchmark story easier to reason about and suggests the packet boundary is stabilizing.

### Once the benchmark subset is cleanly recorded, the next highest-value step may shift away from more packet mechanics
After the ownership-aware benchmark subset is recorded, a documentation/interface pass can have higher project value than immediately widening packet mechanics again.

## Current takeaway
The action boundary is substantially more mature: it now has versioned packet generations, packet cache outcomes, explicit `no_change` behavior, ownership scope/key semantics, baseline and supersede targeting, and a validated ownership-aware benchmark subset whose medium and long representative runs currently favor warmed Numba.
