# 2026 03 29 REV1.7 Model Interface Control Document

## Purpose
Define the boundary between the engine core and the **model developer** role.

## Why this exists now
The current project state has enough validated architecture to make interface control docs worthwhile:
- fallback validation passes
- warmed Numba can outperform NumPy in the integrated comparison path
- rigid-body / gravity / field prototype is returning sane outputs
- AI scheduling and runtime-schema direction are already active architectural truths

This is the right time to formalize **documents and lightweight façades**, but still slightly early for a full repo split.

## Audience
Model developer:
- defines geometry
- defines initial conditions
- defines coordinate systems
- defines actors/archetypes
- defines materials/properties
- defines objectives/scenarios
- owns load/save model state and model validation

## Interface goals
The model developer should be able to:
- build/test model content independently from solver internals
- validate model content before handing it to the engine
- serialize/deserialize scenarios
- reason about object families, coordinate systems, and property payloads
- avoid needing knowledge of physics integrator, AI scheduler, renderer internals, or benchmarking internals

## Engine responsibilities toward this interface
The engine must expose:
1. **Scenario ingest contract**
   - coordinate systems
   - geometry definitions
   - actor/archetype definitions
   - property/material bindings
   - objective definitions
   - initial conditions
2. **Validation contract**
   - schema validation
   - reference consistency checks
   - coordinate-system validity checks
   - property-mask/payload consistency checks
3. **Persistence contract**
   - stable save/load format or adapter layer
4. **Diagnostic contract**
   - import/validation errors
   - entity/coordinate counts
   - model warnings

## Engine internals hidden from the model developer
The model interface should hide:
- SoA field arrangement details
- solver stage ordering
- collision solver internals
- AI batching/scheduling internals
- render implementation details
- backend-specific optimizations

## Primary contracts
### ScenarioBundle
Top-level model handoff into the engine.
Fields conceptually include:
- scenario id
- coordinate system definitions
- geometry assets
- actor assets
- objective assets
- initial conditions

### ModelArtifact
Reusable piece of model content:
- geometry
- materials/property bundles
- actor archetypes
- objectives
- local payloads

### ModelRuntimeView
Engine-to-model-tool view:
- counts
- identifiers
- validation diagnostics
- high-level state summaries

## Ownership boundary
The model developer owns:
- content structure
- semantic meaning of geometry and objectives
- initial state and scenario intent

The engine owns:
- runtime storage
- solver behavior
- scheduling
- performance strategy
- promotion of validated POC internals into shared engine layers

## Required documents that follow from this control doc
- Model Developer Guide
- Scenario/model schema specification
- Save/load format notes
- Validation error catalog

## Near-term implementation guidance
Do now:
- define the contract and the light façade/stub
- define model-side schema vocabulary
- define validation ownership

Defer:
- fully independent modeling application
- stable external package split
- finalized save format

## Risks / cautions
- Do not let model files depend directly on current SoA internals.
- Do not let scenario authoring bypass validation layers.
- Do not freeze final persistence formats too early.
