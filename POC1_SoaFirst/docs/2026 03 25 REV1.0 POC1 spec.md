# pyAIGameEngine Architecture Spec — POC Stack v1 (REV1.0)
**Date:** 2026-03-25  
**Document:** REV1.0 architecture spec (first-choice stack)

## First-choice stack (POC v1)
- **Loop/Window:** `pyglet`
- **Render:** `ModernGL`
- **Physics backend:** `PyBullet`
- **Authoritative state (engine):** NumPy SoA buffers
- **RL API contracts:** Gymnasium + PettingZoo
- **Throughput scaling:** joblib → Ray later
- **Kernel roadmap:** Numba (CPU) → Warp/Taichi (GPU)

---

## 0) Design goals (POC spec)
**Primary goal:** minimize *plank-tic* cycle time while leaving **performance headroom for AI**.

Key architectural commitments:
1. **Data-oriented SoA world state** is the default execution substrate (systems consume arrays/tensors, not Python objects).
2. **PhysicsBackend is swappable** (PyBullet now; kernel backends later).
3. Two execution modes are first-class:
   - **Interactive mode:** single world, stable frame pacing, rendering enabled.
   - **Throughput mode:** many worlds, headless or minimal rendering, multiprocessing.

---

## 1) Core data model: SoA component layouts

### 1.1 Entity identity and component membership
- **EntityId**: `int32` (monotonic counter; never reused within a run unless explicitly recycled)
- **Alive mask**: `alive[N] -> bool`
- **Component bitmask**: `cmask[N] -> uint64` (bit per component)
- **Stable indexing:** entities occupy slots `0..N-1`. Deletion marks slot free; optional compaction is an explicit maintenance task (not per plank-tic).

**Component presence rule:** a system only touches arrays for entities that have required component bits.

---

### 1.2 Physics-related SoA (authoritative “engine state”)
For **N dynamic bodies**:

**Transform / kinematics**
- `pos[N,3] float32`  (x,y,z)
- `rot[N,4] float32`  (quaternion w,x,y,z or x,y,z,w — pick and freeze)
- `linvel[N,3] float32`
- `angvel[N,3] float32`
- `sleep[N] uint8` (0/1) (optional; backend-dependent)

**Mass & inertia (kept simple early)**
- `inv_mass[N] float32`
- `inv_inertia_diag[N,3] float32` (approx; keep diagonal for POC)

**Forces / impulses (accumulators)**
- `force[N,3] float32`
- `torque[N,3] float32`
- `impulse[N,3] float32` (optional)
- `ang_impulse[N,3] float32` (optional)

**Collision filtering**
- `col_group[N] uint32`
- `col_mask[N] uint32`
- `restitution[N] float32`
- `friction[N] float32`

**Shape**
- `shape_type[N] uint8`  
  - 0 sphere, 1 box, 2 capsule, 3 mesh (later), 4 heightfield (later), …
- `shape_param[N,4] float32`  
  - sphere: (r,0,0,0)  
  - box: (hx,hy,hz,0)  
  - capsule: (r, half_height, 0, 0)

**Static bodies**
Keep a separate store for static colliders:
- `static_pos[M,3]`, `static_rot[M,4]`, `static_shape_type[M]`, `static_shape_param[M,4]`, `static_col_group[M]`, `static_col_mask[M]`

This avoids mixing “rarely-changing” static geometry with hot dynamic arrays.

---

### 1.3 AI components (SoA + small metadata handles)

**Agent membership**
- `agent_active[N] uint8` (1 if entity is an agent-controlled body)
- `agent_id[N] int32` (maps entity slot → agent index; -1 if none)
- `policy_id[N] int16` (which policy/controller instance to use)
- `team_id[N] int16` (for MARL grouping)
- `reward_accum[N] float32` (or per-agent array)

**Action buffer (written by policies, read by simulation/gameplay)**
For POC, keep generic continuous action:
- `act[N, A] float32` where `A` is max action width
- `act_mask[N, A] uint8` (optional: which dims valid)
- For discrete or hybrid: store a tagged union in arrays:
  - `act_type[N] uint8` (0 cont, 1 discrete, 2 hybrid)
  - `act_discrete[N] int32`
  - `act_cont[N,A] float32`

**Observation views**
Avoid per-agent Python dicts in the hot path. Use:
- `obs_buf[Agents, ObsDim] float32` (contiguous)
- `obs_index[Agents] int32` (maps agent → entity slot)
- Multi-modal later: `obs_dict` is a *view* built at the wrapper layer, not the internal substrate.

---

### 1.4 Render components (decoupled from physics truth)
Rendering consumes **snapshots**. Keep the minimum instance attributes:
- `render_enabled[N] uint8`
- `mesh_id[N] int32`
- `material_id[N] int32`
- `scale[N,3] float32`
- `color[N,4] float32` (optional)
- `xform44_snapshot[K,16] float32` (packed transforms uploaded once per frame, where K = rendered entities)

> Rule: renderer never reads “live mutable” simulation arrays mid-step; it reads a snapshot.

## 2) PhysicsBackend interface + PyBullet implementation boundaries

### 2.1 Backend contract
The PhysicsBackend exists to:
- own backend-specific state
- advance simulation
- provide contact/query results
- synchronize with SoA buffers at explicit boundaries

Python-ish interface (spec):

```python
class PhysicsBackend(Protocol):
    name: str
    device: str  # "cpu" | "cuda" | ...

    def create_world(self, world_desc: dict) -> None: ...
    def destroy_world(self) -> None: ...

    # entity lifecycle
    def add_body(self, eid: int, body_desc: dict) -> None: ...
    def remove_body(self, eid: int) -> None: ...
    def add_static(self, sid: int, static_desc: dict) -> None: ...

    # sync boundaries
    def sync_in(self, state: "SoAState", dirty: "DirtyFlags") -> None: ...
    def step(self, dt: float, substeps: int) -> None: ...
    def sync_out(self, state: "SoAState", what: "SyncMask") -> None: ...

    # queries/events
    def get_contacts(self, out: "ContactBuffer", filter: dict) -> int: ...
    def raycast(self, origin, direction, max_dist, mask) -> "RayHit": ...
    def aabb_query(self, aabb_min, aabb_max, mask) -> "IndexList": ...
```

**DirtyFlags** example: positions changed, kinematic targets changed, forces changed, shape changed (rare), etc.

---

### 2.2 PyBulletBackend: exact boundaries (what lives where)
**PyBulletBackend responsibilities**
- Maintain mapping: `eid -> bullet_body_id`
- Own Bullet “world” config: gravity, solver iterations, ERP/CFM-ish params, substeps
- Create Bullet collision shapes and bodies once (avoid per-tick creation)
- During `sync_in`:
  - apply forces/torques
  - apply kinematic targets (if you support kinematic bodies)
  - optionally teleport bodies when `pos/rot` dirty
- During `step`:
  - call Bullet stepSimulation
- During `sync_out`:
  - pull `pos/rot`, `linvel/angvel` for requested subset
  - optionally pull contact data (but keep it opt-in; it’s expensive)

**Engine responsibilities**
- Maintain authoritative SoA (even if currently filled from Bullet)
- Decide *when* to sync and *what* to sync (SyncMask)
- Maintain per-system invariants (double-buffering/snapshots)
- Translate events into gameplay/AI signals

**Important rule for POC v1**
- For simplicity, treat Bullet as the solver-of-record, but **the engine arrays are the canonical interface**.
  - In PyBullet backend, canonical arrays are updated by `sync_out`.
  - In future kernel backends, canonical arrays are updated by kernels directly.

This keeps *all* higher layers stable while physics implementations change.

## 3) Task graph / scheduling model (interactive vs throughput)

### 3.1 Canonical stage pipeline (per plank-tic)
We define a fixed high-level stage order; each stage may expand into tasks.

**Stage order**
1. **Input + external commands** (spawn/despawn, user inputs)
2. **AI inference** (policies/BTs write actions)
3. **Gameplay pre-physics** (apply action → forces/targets)
4. **Physics** (backend sync_in → step → sync_out)
5. **Gameplay post-physics** (contacts → events → rewards/termination)
6. **Observation build** (for next step)
7. **Render snapshot build**
8. **Render** (if interactive)

ASCII task graph:

```text
[Commands/Input]
      |
      v
 [AI Inference] ---> (optional async policy workers)
      |
      v
[Apply Actions -> Forces/Targets]
      |
      v
[PhysicsBackend.sync_in] -> [PhysicsBackend.step] -> [PhysicsBackend.sync_out]
      |
      v
[Contacts/Events -> Rewards/Done]
      |
      v
[Build Observations] --> (export to Gym/PZ wrappers)
      |
      +--> [Build Render Snapshot] --> [Render]
```

---

### 3.2 Interactive mode (single process, stable frame pacing)
**Constraints**
- pyglet’s event loop wants to own the main thread (practically).
- Python threads won’t accelerate CPU physics math; they *can* overlap I/O or native calls.

**Model**
- Main thread runs: pyglet event loop + engine step at fixed dt (or accumulator).
- Optional background thread(s):
  - logging, metrics, asset streaming
  - AI inference *only if* it’s in native code and releases GIL (e.g., Torch) and you can tolerate 1-frame delay.

**Scheduling**
- Deterministic: fixed stage order, no multi-process.
- Physics runs in-process (PyBullet calls).

---

### 3.3 Throughput mode (multi-process “world workers”)
**Goal:** maximize steps/sec across many independent environments.

**Model**
- **Coordinator process**:
  - spawns worker processes
  - aggregates rollouts
  - trains policy (SB3/PyTorch) or coordinates learners
- **Worker process**:
  - holds 1..B worlds (batch) headless
  - runs step loop with minimal Python overhead
  - emits rollout chunks (obs, act, rew, done, info)

**Worker parallelism choices**
- **POC2:** `joblib` for local multiprocessing
- **Later:** Ray actors when scaling/elasticity matters

**Data exchange**
- Prefer shared memory for rollout arrays (later).
- POC can use pickle for small batches; but the roadmap should push toward shared memory buffers quickly.

## 4) Observation/action pipeline (Gymnasium + PettingZoo)

### 4.1 Engine-internal API (stable, fast)
Engine exposes two pure-array views:
- `ObservationView`: “read-only” view of SoA buffers for a given agent batch
- `ActionBuffer`: writeable action arrays

```python
class ObservationView:
    # returns contiguous float32 array [num_agents, obs_dim]
    def build_obs(self, agent_index: np.ndarray, out: np.ndarray) -> np.ndarray: ...

class ActionBuffer:
    # actions written by policies; applied by gameplay system
    act: np.ndarray  # [num_agents, act_dim], float32
```

### 4.2 Gymnasium wrapper (single-agent or vectorized)
Gymnasium environment uses `step(action)` and `reset()`

**Single-agent**
- `action` targets the single agent entity
- returns `(obs, reward, terminated, truncated, info)`

**Vectorized env (recommended early for throughput)**
- Use Gymnasium `VectorEnv` / `AsyncVectorEnv` (or your own wrapper)
- Each env maps to one World instance in a worker
- Observations are stacked `[num_envs, obs_dim]`

### 4.3 PettingZoo wrapper (multi-agent)
PettingZoo provides multi-agent API; prefer **Parallel API** for performance

**Parallel step**
- input: `actions: Dict[agent_name, action]`
- output: `obs, rewards, terminations, truncations, infos` dicts

**Performance trick**
Internally keep array form:
- `agent_names = ["a0","a1",...]`
- `name_to_index` mapping
- Convert dict→array at boundary only; engine stays array-based.

## 5) Kernel backend roadmap (what migrates first to Numba/Warp/Taichi)

### 5.1 Why this order
You migrate stages that:
- are **hot**,
- are **local** (few dependencies),
- have **clear data parallelism**,
- and won’t force you to rebuild the world model too early.

---

### 5.2 Migration phases (stages → backends)

**Phase A (POC2): Numba CPU kernels**
1. **Integrate (semi-implicit Euler)** for entities not driven by PyBullet (particles, simple kinematics)
2. **Broadphase acceleration** (uniform grid / spatial hashing) for simple primitives
3. **Observation building** (large batched feature extraction) — often a hidden bottleneck

**Phase B (POC3): Warp or Taichi GPU kernels**
4. **Broadphase grid build + pair generation** on GPU  
5. **Narrowphase** for primitive-primitive contacts (sphere-sphere, sphere-box, box-box approx)
6. **Contact reduction + manifold building** (keep minimal early)

**Phase C (later): GPU solver**
7. **XPBD / PBD constraint solve** (island batching)  
8. **Articulations/joints** (hard; keep in PyBullet or switch to a backend designed for it)

> Practical stance: **do not** try to “replace Bullet” in one shot. Replace *stages*.

## 6) CPU ↔ GPU transfer policy (explicit decisions)

### 6.1 Rule set (v1)
We define “device locality” per subsystem. A subsystem may be CPU or GPU resident.

**Allowed transfers (v1)**
1. **CPU sim → GPU render upload** (interactive mode only)  
   - Allowed once per rendered frame, via a single packed instance buffer.
2. **CPU obs → GPU policy inference** (optional)  
   - Allowed, but flagged as “throughput risk” for high-frequency stepping.
3. **GPU sim (future) → CPU logging/termination**  
   - Allowed only for *small* metadata (done flags, scalar rewards), or batched snapshots at coarse intervals.

**Forbidden / discouraged**
- Per-entity per-step CPU↔GPU ping-pong (e.g., “simulate on GPU, pull full state to CPU every tick for AI, then push actions back”).
- Rendering reading live simulation buffers mid-step (must use snapshots).

### 6.2 Device ownership plan
- **POC1–POC2:** physics on CPU, observations on CPU, rendering on GPU (uploads OK)
- **POC3:** introduce optional GPU-resident observation builder and/or primitive physics kernels; keep AI inference co-located with observation data when possible (GPU→GPU path)

## 8) Dataflow diagrams

### 8.1 Interactive mode (single world)
```text
pyglet event loop (main thread)
  |
  +-- fixed-step accumulator --> for each plank-tic:
         Commands/Input
           -> AI (optional)
           -> Apply actions -> forces/targets (SoA)
           -> PhysicsBackend.sync_in(SoA)
           -> PhysicsBackend.step(dt)
           -> PhysicsBackend.sync_out(SoA)
           -> Events/Rewards/Done
           -> Build Observations
           -> Build Render Snapshot (packed transforms)
  |
  +-- per frame: Renderer uploads snapshot buffer -> draw
```

### 8.2 Throughput mode (many worlds)
```text
Coordinator process:
  policy/learner (PyTorch/SB3)  <--- rollout batches ---  Workers (N processes)
           |                                           each worker:
           +--- broadcast weights --->                 step loop (headless)
                                                       emit obs/act/rew/done chunks
```

## 9) Milestone plan: three POC iterations

### POC Iteration 1 — “Single world, stable loop, backend boundary”
**Deliverables**
- SoAState buffers + component masks
- PhysicsBackend + PyBulletBackend working for spheres/boxes
- pyglet + ModernGL viewer rendering instanced primitives
- Deterministic fixed-step plank-tic loop (basic)
- Minimal contact events (ground hit) and debug overlay

**Success metric**
- smooth real-time stepping + visible physics + clean backend interface

---

### POC Iteration 2 — “Throughput mode + Gymnasium vectorization”
**Deliverables**
- Gymnasium wrapper + vectorized env stepping
- joblib multiprocessing world workers
- Rollout buffer format (contiguous arrays)
- Baseline SB3 training on a simple task (cartpole-like but 3D-ish)
- Numba kernels for observation building (first real speed win)

**Success metric**
- measurable steps/sec scaling with CPU cores; training loop produces improving reward

---

### POC Iteration 3 — “Kernel beachhead: GPU broadphase (Warp or Taichi)”
**Deliverables**
- Add `KernelBackend` concept (not replacing PhysicsBackend yet)
- GPU broadphase grid build + pair gen on Warp *or* Taichi
- Optional primitive narrowphase for spheres
- Define device-transfer policy enforcement (assertions + counters)
- PettingZoo parallel wrapper for a tiny MARL demo

**Success metric**
- stage-level GPU acceleration demonstrated without destabilizing the engine API

---

## 10) Explicit “don’t build from scratch yet” decisions (v1 spec)
1. **Constraint solver / full rigid body engine** → use PyBullet for now  
2. **Full editor UI** → use lightweight tooling (later DearPyGui/ImGui); keep engine-core clean  
3. **Asset formats** → use glTF + external DCC (Blender) rather than inventing formats  
4. **Distributed runtime** → joblib now, Ray later; don’t write your own scheduler

## 11) ADR-style decisions (concise)
**ADR-001: Authoritative state is SoA arrays**  
Status: Accepted (POC)  
Decision: Systems operate on SoA buffers; Python objects are handles only.

**ADR-002: PhysicsBackend is swappable; PyBullet is bootstrap**  
Status: Accepted (POC)  
Decision: Use PyBullet behind a strict sync boundary; design for kernel migration.

**ADR-003: Two execution modes**  
Status: Accepted (POC)  
Decision: interactive single-process + throughput multiprocessing are both supported from the start.

**ADR-004: Observation/action pipeline is array-native; wrappers adapt to Gym/PZ**  
Status: Accepted (POC)  
Decision: dict-based APIs exist only at wrapper edges.

**ADR-005: Explicit device transfer policy**  
Status: Accepted (POC)  
Decision: transfers only at declared boundaries; track/counter transfers.
